# PKM 框架整合地圖 — 構建多維度的數位治理體系

> [!abstract] 核心洞見 (Insight)
> **框架不是信仰，而是解決特定問題的工具。** PKM 框架整合地圖的核心定位是系統的 **「方法論地圖」**。當代主流框架並非互斥，而是服務於知識生命週期的不同維度。PARA 負責 **「在哪裡執行 (Where)」**，CODE 負責 **「如何流動 (How)」**，Zettelkasten 負責 **「如何產生洞察 (Why)」**。在 H3.0 系統中，我們的目標不是選邊站隊，而是透過多代理（Multi-Agent）協調，讓這些框架在同一系統中實現 **「軸轂協同 (Hub-and-Spoke)」**。**整合的本質是將「行動力」與「深度思考」進行無縫接力。**

### 定義與機制 (What & How)

#### 框架職能對照表 (The PKM Matrix)

| 框架 | 核心邏輯 (Logic) | 組織原則 | 適用場景 | 系統角色 |
| :--- | :--- | :--- | :--- | :--- |
| **PARA** | 行動導向 (Action) | P(燃燒) / A(灌溉) / R(圖書館) | 專案執行、任務流管理 | 路由器 (Router) |
| **CODE** | 知識演化 (Flow) | Capture → Organize → Distill → Express | 內容產出、寫作工作流 | 蒸餾器 (Distiller) |
| **Zettelkasten** | 原子連結 (Insight) | 唯一標識符 + 雙向連結 | 深度研究、創意湧現 | 連結器 (Linker) |
| **ACCESS/ACE** | 維度映射 (Space) | Atlas(空間) / Calendar(時間) / Efforts(行動) | 上下文切換、能量管理 | 平衡者 (Balancer) |

#### 核心協同模型 (Integration Models)

##### 1. PARA × CODE：執行引擎
PARA 定義「存放座標」，CODE 定義「加工流程」。
- **Capture** ⮕ **INBOX**
- **Organize** ⮕ **PARA 分流**
- **Distill** ⮕ **漸進式摘要** (提取特定知識)
- **Express** ⮕ **產出** (專案計畫/文案)

##### 2. Zettelkasten × PARA：軸轂架構 (Hub-and-Spoke)
- **PARA (前線辦公室)**：處理動態、高頻、專案導向的資訊。
- **卡片盒 (研究圖書館)**：存放穩定、原子化、長效的「特定知識」。
- **協同機制**：當專案（P）結束時，系統架構員 Agent 主動將專案中的精華洞見「脫水」並移至卡片盒，實現知識資產化。

#### 框架協同矩陣 (Synergy Matrix)
- **PARA + CODE** ⮕ 分層目錄 + 自動化管道 (由路由器代理驅動)。
- **Zettelkasten + ACCESS** ⮕ 原子卡 + MoC 彙整 (由連結代理驅動)。
- **全部整合** ⮕ **Hub-and-Spoke 軸轂架構** (由協調者 Orchestrator 驅動)。

### 實踐與案例 (Action & Proof)
- **真實案例**：Navi Dashboard 系統設計中的「多框架協作」。
  - **背景**：需要處理複雜的 Boolean 代碼邏輯，同時要管理 UX 設計進度與多平台分發。
  - **執行流程**：
    1. 使用 **PARA** 管理專案進度（P）與資源（R）。
    2. 使用 **CODE** 從多個技術文庫中蒸餾（Distill）出核心代碼段。
    3. 使用 **Zettelkasten** 將「Boolean 邏輯」與「UX 心理學模式」進行跨領域連結，產生獨特的設計洞見。
  - **成果**：專案不僅如期完成，且產出的「設計規範」被固化為長效知識資產。**整合框架讓「執行」與「思考」不再脫節。**
- **行動指南**：
  1. **不要為分類而分類**：如果一個檔案不知道放哪，先丟進 **INBOX**，讓系統根據 **CODE** 流程在當週週末進行分流。
  2. **強化 MoC (Map of Content)**：使用 ACCESS 框架中的 Atlas 概念，為每個核心領域建立地圖，作為 PARA 與 Zettelkasten 的交匯點。
