## 層級一：000捕獲（Inbox）

### 功能定位
系統唯一的快速輸入口，用於卸載大腦中的即時想法、資訊碎片、臨時任務。

### 核心問題
**這是黃金還是垃圾？**

### 運作規則

**規則1：7天處置鐵律**
- 任何進入 000捕獲 的項目必須在7天內處理
- 第8天自動視為「確認無價值」，直接刪除
- 無例外情況

**規則2：處理路徑決策樹**
```
捕獲項目 → 判斷：
├─ 能讓我行動更快？        → 300知識
├─ 是某個專案的具體任務？   → 001執行層
├─ 能成為創作素材？         → 400創作區
├─ 是長期願景或反思？       → 100夢想
└─ 以上都不是              → 刪除
```

**規則3：禁止在此處整理**
- 000捕獲 不進行任何加工、分類、整理
- 僅做「轉移」或「刪除」決策
- 避免將 000 變成第二個工作區

### 關鍵檔案說明

**_捕獲待處理清單.md**
- 功能：自動列出所有超過5天未處理的項目
- 更新頻率：每日檢視
- 作用：預警機制，避免遺漏

**命名規則**
- 捕獲項目檔名格式：`YYYYMMDD-簡短描述.md`
- 範例：`20250107-產品需求想法.md`
- 日期戳記用於自動計算停留天數

### 與其他層級的連結

| 目標層級 | 連結機制 | 範例 |
|---------|---------|------|
| 300知識 | 移動檔案 + 加入知識池連結 | `[[300知識/310專案知識/專案A知識池]]` |
| 001執行層 | 轉化為任務項目 + 設定到期日 | 從捕獲項目提取行動，寫入今日作戰 |
| 400創作區 | 移動至靈感池 | `[[400創作區/410靈感池/想法X]]` |
| 100夢想 | 移動至相應日誌 | `[[100夢想/120日誌/2025-01-07]]` |

## 層級二：090 KPI儀表板（Dashboard）

### 功能定位
成果可視化中心，追蹤12週內的三種核心成果指標。**不具有否決權，僅作為結果追蹤工具。**

### 核心問題
**這12週我產出了什麼可證明的成果？**

### 三種KPI類型詳細說明

#### KPI類型1：現金流指標（Cash Flow Metrics）

**定義**
- 任何形式的金錢收入
- 包含：產品銷售、服務收費、廣告分潤、贊助收入、課程收入

**測量方式**
- 單位：貨幣金額（新台幣/美元）
- 追蹤頻率：每週更新
- 記錄內容：金額、來源、日期

**範例目標設定**
```
目標：本12週產生 $10,000 收入
週度分解：
- W1-W4：$2,000（產品開發期，低收入）
- W5-W8：$4,000（產品上線，開始銷售）
- W9-W12：$4,000（優化銷售，穩定收入）
```

#### KPI類型2：影響力指標（Influence Metrics）

**定義**
- 能證明「有人因你而改變」的量化指標
- 包含：訂閱數、追蹤數、轉發數、邀約數、推薦數

**測量方式**
- 單位：人數、次數
- 追蹤頻率：每週更新
- 記錄內容：數量、來源平台、內容類型

**範例目標設定**
```
目標：本12週獲得 500 新訂閱
週度分解：
- W1-W4：100人（初期建立內容，慢速增長）
- W5-W8：200人（內容累積，加速增長）
- W9-W12：200人（穩定產出，持續增長）
```

#### KPI類型3：能力建設指標（Capability Building Metrics）

**定義**
- 長期投資型活動的成果，短期內不直接產生現金或影響力
- 包含：完成的知識體系數、掌握的技能數、閱讀的專業文獻數、完成的研究階段

**測量方式**
- 單位：主題數、技能數、論文數、里程碑數
- 追蹤頻率：每週更新
- 記錄內容：完成的具體項目名稱、完成標準

**範例目標設定**
```
目標：本12週完成 3 個主題的知識體系建構
週度分解：
- W1-W4：主題1完成（閱讀10篇論文 + 整理框架）
- W5-W8：主題2完成（閱讀10篇論文 + 整理框架）
- W9-W12：主題3完成（閱讀10篇論文 + 整理框架）
```

**為何需要此類型KPI**
- 保護學術研究、基礎學習等長期投資活動
- 避免純功利主義導向（不是所有活動都能立即變現）
- 但仍需量化追蹤，確保「投資」有產出

## 層級三：010目標控制層（Goal Control Layer）

### 功能定位
整個系統的戰略中樞，設定12週內的核心目標，提供所有行動的權力來源。

### 核心問題
**這12週結束時，什麼成果能證明我沒有浪費這段時間？**

### 12週循環完整說明

#### 為何選擇12週？

**理論依據：《12週做完一年工作》（The 12 Week Year）**
- 作者：Brian P. Moran, Michael Lennington
- 核心概念：將一年視為四個12週，每個12週視為一個完整年度
- 心理機制：12週足夠長（可完成有意義的目標），又足夠短（保持緊迫感）
- 對比傳統年度目標：避免「還有時間」的拖延心態

**12週 vs 其他時間週期**

| 時間週期 | 優點 | 缺點 | 適用情境 |
|---------|------|------|---------|
| 1個月 | 緊迫感強 | 無法完成複雜目標 | 短期衝刺 |
| 1季（3個月） | 接近12週 | 心理上仍偏長 | 企業季度目標 |
| 半年 | 可完成大型目標 | 拖延感嚴重 | 長期研究 |
| 1年 | 完整性高 | 執行力低落 | 願景規劃 |
| **12週** | **平衡緊迫感與可行性** | 需頻繁規劃 | **執行力導向系統** |

#### 12週循環的四個階段

**階段1：規劃週（Week 0）**
- 時間：12週開始前的週末
- 任務：設定3-4個核心目標、建立專案儀表板、設定KPI
- 產出：當前12週.md 完成

**階段2：執行期（Week 1-10）**
- 時間：前10週
- 任務：每週提取任務、每日執行、每週回顧
- 產出：每週計分卡完成

**階段3：衝刺期（Week 11-12）**
- 時間：最後2週
- 任務：聚焦核心目標、放棄低價值任務、全力達成KPI
- 產出：目標達成或明確的未達成原因

**階段4：復盤週（Week 13）**
- 時間：12週結束後的週末
- 任務：評估目標達成率、知識庫清理、系統優化、規劃下個12週
- 產出：12週復盤.md 完成

## 層級四：001執行層（Execution Layer）

### 功能定位
系統的唯一行動入口，所有待辦任務統一在此處理。**集權化任務管理。**

### 核心問題
**今天的3個「不可失敗」完成了嗎？**

### 核心概念：MITs (Most Important Tasks)

**定義**
- Most Important Tasks 最重要任務
- 每日最多3個
- 完成這3個任務後，今天即使完成其他0個任務也算成功

**與一般待辦清單的差異**

| 維度 | 一般待辦清單 | MITs系統 |
|------|------------|---------|
| 任務數量 | 無上限 | 嚴格限制3個 |
| 優先級 | 多級別（高/中/低） | 只有「不可失敗」一種 |
| 完成標準 | 模糊 | 明確定義 |
| 未完成處理 | 自動延期 | 強制解釋 |
| 心理負擔 | 看到長清單焦慮 | 只需專注3件事 |

### 關鍵機制：任務到期日+解釋義務

#### 機制1：強制到期日（Mandatory Deadline）

**規則**
- 所有任務必須有明確到期日
- 格式：YYYY-MM-DD
- 無到期日的任務不允許存在

**到期日設定邏輯**

| 任務類型 | 到期日設定方式 | 範例 |
|---------|---------------|------|
| 來自週計分卡的任務 | 本週最後一天（週日） | 2025-01-12 |
| 突發緊急任務 | 當日或次日 | 2025-01-08 |
| 長期任務 | 拆解為多個短期任務，每個都有到期日 | 週次拆解 |
| 等待他人的任務 | 設定追蹤日（非完成日） | 2025-01-15（追蹤日）|

**到期日的心理機制**
- 帕金森定律（Parkinson's Law）：工作會膨脹至填滿所有可用時間
- 有到期日 → 產生時間壓力 → 提升執行效率
- 無到期日 → 無時間壓力 → 無限延後

#### 機制2：解釋義務（Accountability Obligation）

**觸發條件**
- 任何MIT未在到期日前完成
- 任何任務超過到期日

**解釋義務四選一**
```
未完成任務必須回答以下其中一項：

□ 目標錯誤
  說明：這件事其實不值得做，因為___
  處理：刪除任務，可能需要調整上游目標

□ 任務拆解不足
  說明：這個任務實際需要___小時，而非預估的___小時
  處理：拆解為更小的子任務，重新設定到期日

□ 外部干擾
  說明：主要干擾因素是___
  處理：下次如何隔離時間？需要什麼條件？

□ 能力不足
  說明：完成此任務需要___能力，我目前不具備
  處理：需要學習什麼？向誰求助？還是放棄此目標？
```

**解釋義務的作用**
- 防止自我欺騙（不允許沈默延期）
- 系統優化（累積數據識別常見障礙）
- 目標修正（發現錯誤目標時及時止損）

**解釋義務的記錄方式**
```
# 001執行層/過期任務追蹤.md

## 本週過期任務

### [任務X]（過期3天）
- 原定到期日：2025-01-07
- 原始目標：完成產品原型設計
- 解釋類型：☑ 任務拆解不足
- 詳細說明：
  原預估需要8小時，實際需要20小時。
  原因：低估了技術難度，API文件不完整需額外研究。
- 處理決策：
  - 拆解為3個子任務：
    1. 研究API文件（4h，到期日：2025-01-08）
    2. 完成核心功能（10h，到期日：2025-01-10）
    3. 完成UI整合（6h，到期日：2025-01-12）
