# 資料清理與整合防護協議 (Data Safety & Cleanup Protocol)

> [!CAUTION]
> 為了防止資料在「原子化」或「重組」過程中永久遺失，AI 必須嚴格遵守此協議。備份只是最後手段，**流程防護**才是核心。

## 1. 執行前置準則 (Pre-Execution Principles)

### 1.1 步驟極致拆解 (Atomic Step Decomposition)
- **禁止合併指令**：不可在一個步驟中同時執行「寫入新檔」與「刪除舊檔」。
- **任務隔離**：每一個資料夾的遷移必須是一個獨立的 `TaskName`。

### 1.2 資料保護優先
- **內容召回保險**：在刪除任何舊檔案前，AI 必須已讀取並記錄該檔案的核心內容至當前對話上下文。

## 2. 三階段強制確認流程 (The Triple-Check Workflow)

### 階段 I：預檢 (Pre-check)
1. **建立顯式清單**：羅列所有預計新增的目標檔案，以及預計刪除的來源檔案路徑。
2. **原子化路徑映射**：明確標註「資料 A 從檔案 X 搬移至檔案 Y」。

### 階段 II：驗證執行 (Verifiable Execution)
1. **先建後刪**：必須先完成所有新筆記的建立 (Write)。
2. **存在性校驗**：使用 `find_by_name` 或 `view_file` 確保新筆記已正確寫入且包含應有的內容。
3. **連結追蹤**：確保新筆記已掛載至對應的 MOC。

### 階段 III：獲准刪除 (Authorization to Delete)
1. **顯式清單呈報**：將「核對完成」後的刪除清單呈報給使用者。
2. **逐項批准**：使用者必須針對刪除清單給予明確的批准訊號（如 "LGTM" 或 "Proceed with deletions"）。
3. **禁止批次模糊刪除**：嚴禁執行 `rm -rf` 等模糊路徑刪除命命。

## 3. 防護失敗的補救措施 (Remediation)

- 若發現連結斷裂 (404)，AI 必須立即搜尋過往 Session 紀錄進行內容重建。
- 若使用者要求比對變更，AI 必須提供 `changes_summary.md` 格式的詳細對照表。

## 4. 系統性防護指令 (Safety Commands)

當使用者輸入以下關鍵字時，AI 必須強制進入防護模式：
- `//CLEANUP`：啟動此協議。
- `//MIGRATE`：啟動遷移防護。
- `//RESTORE`：啟動資料找回流程。
