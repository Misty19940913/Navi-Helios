# YAML 屬性建置規則

## 一、核心原則

- **YAML 最小化**：屬性僅保留功能性調度（如：象限、排程）。
- **標籤分類化**：採用 `分類/值` 格式（如：`系統/任務`），避免全域標籤污染。
- **查詢優化**：所有屬性設計均為了 DataviewJS 的高效篩選。

## 二、標準 YAML 結構（7行核心）

```yaml
---
tags: [系統/xx] # 標籤分類系統
status: draft # 狀態
time_create: YYYY-MM-DD # 建立日期 (由插件自動填充)
time_modifie: YYYY-MM-DD # 修改日期 (由插件自動填充)
subject: [[範疇MOC]] # 核心連結：跨域範疇、主題
  - [[範疇MOC]]
parent: [[上層連結]] # 權力溯源：指向所屬專案或夢想(選填)
related: [[水平連結]] # 水平關聯筆記（選填，不具上下層級關係）
---
```

### 3.2 特殊情況保留的 YAML 屬性

#### 分析導向屬性（HUMAN 3.0）

- `quadrant`: "心智" | "身體" | "精神" | "職業"（HUMAN 3.0 生命領域）
- HUMAN 3.0 生命領域
	- **心智**：個人心理世界、思考、覺醒。
	- **身體**：個人物質世界、健康、生存。
	- **精神**：集體心理世界、社交、使命。
	- **職業**：集體物質世界、事業、財務。

#### 3.3 行動導向屬性 (Action-Oriented)

**適用：#系統/任務、#系統/專案**
- `parent`: [[連結]]（垂直權力溯源，必填）
- `scheduled`: YYYY-MM-DD（排程執行日期）
- `fail-reason`: string（失敗原因，僅在異常中止時填寫）

#### 3.5 狀態定義 (Status)

**【系統通用與任務狀態】**
這主要運用於任務(`230任務池`)與專案領域(`300專案`)。
| **狀態 (Value)** | **狀況**                |
| -------------- | --------------------- |
| **`todo`**     | 排程日期還沒到、準備執行的專案。 |
| **`waiting`**  | 專案卡住、等回覆、擱置中。 |
| **`doing`**    | 正在寫、正在執行、正在研究。 |
| **`done`**     | 專案發布了、任務完成了。 |
| **`canceled`** | 取消 / 放棄 |
| **`archived`** | 歸檔 |

**【數位花園：純知識成熟度狀態】**
這專門運用於提煉後的原子化知識點(`400知識`)。
| **狀態 (Value)** | **狀況**                |
| -------------- | --------------------- |
| **`seed`**     | （知識種子）剛擷取的靈感、初級碎片，尚未消化。 |
| **`in-progress`** | （培育中）正在被某個「進行中專案」調用或研究的知識點。 |
| **`evergreen`** | （長青知識）已經高度吸收內化，可隨時被新專案調用的核心知識積木。 |

### 四、使用原則與 Dataview 範例

## 使用原則

1. **新檔案**：直接使用標準 YAML 結構
2. **關聯建立**：若看到 A 筆記與 B 筆記有啟發關係，但在權力上互不隸屬，請將其放入 `related`。
3. **驗證檢查**：用 Dataview 查詢驗證遷移結果[[Dataview 查詢範例]]
