# 協議：專案進度分析 (Project Progress Analysis)

## 0. 啟動條件 (Triggers)
- 當使用者輸入 `//status` 或要求「專案進度」、「更新狀況」時。
- 當進入新的週期（週/月）需要對 300 區專案進行結算時。

## 1. 執行程序 (SOP)

### 第一階段：資訊擷取
1. **目錄定位**：鎖定 `300專案/` 下對應的子目錄。
2. **核心讀取**：優先讀取 `README.md`、`Index.md` 或最近的進度日誌。
3. **情境對齊**：確認該專案與 `100夢想/` 層級的連結關係。

### 第二階段：結構化診斷
將雜亂的進度資訊轉化為以下三類：
- **已達成 (Accomplished)**：已完成的關鍵里程碑。
- **推進中 (Active)**：目前正在佔用認知資源的任務。
- **待啟動 (Backlog)**：計畫中但尚未投入資源的項目。

## 2. 輸出範本 (Output Template)

```markdown
## 📁 專案狀態摘要：[專案名稱]

**📊 當前進度分佈：**
- ✅ **已完成**：(列出 2-3 項)
- ⏳ **進行中**：(列出 2-3 項)
- 📅 **待啟動**：(列出 2-3 項)

**🚀 建議下一步 (Next Actions)：**
1. (根據 Readme 邏輯建議的動作 1)
2. (建議的動作 2)
3. (建議的動作 3)
```

---
**僅回傳摘要與建議。輸出的下一步行動需符合「原子化」原則，且能直接被 [[原子任務捕獲協議]] 處理。**
