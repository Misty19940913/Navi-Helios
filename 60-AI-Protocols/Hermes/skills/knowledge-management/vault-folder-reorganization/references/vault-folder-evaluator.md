# Vault Folder Evaluator

## 觸發條件

當需要評估某個 vault 資料夾的價值時觸發——無論是要刪除、蒸餾、還是單純確認是否有在使用。

典型句式：
- 「你確認一下 `XXX` 資料夾有沒有在使用？」
- 「這個資料夾對你有沒有價值？」
- 「你有去閱讀過嗎？」（用戶已察覺你只列了檔名沒讀內容）

## 核心原則

> **列出檔名 ≠ 讀過內容。沒有讀內容的評估 = 猜測。**

過去犯過的錯誤：
- 只用 `ls` 列出 `60_Shared/` 的檔名，就說「我沒有在使用」
- 沒讀 `_misty-model.md` 的內容，就說它「對我沒價值」
- 沒讀 `670提示詞/` 的內容，就說「建議刪除」

這些錯誤都被用戶當場抓包。

## 工作流

### Step 1 — 列出資料夾內容

```bash
find "<vault-path>/<folder>/" -type f -name "*.md" | sort
```

### Step 2 — 讀取 index.md（如果存在）

如果資料夾有 `index.md` 或 `MOC.md`，先讀這個。結構定義是理解內容用途的第一步。

### Step 3 — 讀取核心內容檔案（禁止跳過）

對於每個子資料夾或核心檔案，至少讀一個實質內容檔：

| 檔案類型 | 最少讀取量 |
|---------|----------|
| 設定/結構檔（如 `_schema.md`） | 完整讀取 |
| 主要內容（如 `SKILL.md`） | 完整讀取 |
| 示範/範例檔 | 隨機抽 1-2 個 |
| 設定檔（如 `settings.json`） | 跳過（無內容）|

**如果不知道哪個是最重要的檔案，就全部讀。**

### Step 4 — 查詢對話歷史（如果有）

如果資料夾有明確的歷史脈絡，用 `session_search` 查詢相關對話：

```python
session_search("資料夾名 OR 檔案名 OR 專案名")
```

過去的對話可能已經有結論——不需要重新猜測。

### Step 5 — 輸出評估結果

```
## 評估：{資料夾名稱}

### 資料夾定位
（一句話說明這是什麼資料夾）

### 子資料夾/檔案分析

| 項目 | 內容摘要 | 對我的價值 |
|------|---------|-----------|
| ... | ... | 高/中/低/無 |

### 關鍵發現
（如果有的話——例如：內容與我想的不同、有認知盲區、有重複資訊）

### 建議
- 刪除：（如果真的可以刪）
- 蒸餾：（如果要保留核心價值）
- 保留：（如果對 Misty 有價值）
```

### Step 6 — 如用戶要求「你去讀過了嗎」

立刻承認：「沒有，我剛才只列了檔名，現在補讀。」然後立即執行 Step 2-3。

## 整合搬遷工作流（適用於 60_Shared → 新位置）

當要把一個資料夾的內容整合進現有系統時：

1. **先建新目錄結構**（`mkdir -p`）
2. **逐步搬遷每個子資料夾**，每個完成後立即驗證
3. **每個階段結束時檢查「還剩什麼」**——防止結構定義檔（index.md/_schema.md）被遺漏
4. **比較同名檔案**：`wc -l` 比大小，內容更完整的留下並更新進階版本
5. **全部確認後再刪除原資料夾**（「先建後刪鐵律」）
6. **如果原資料夾刪不掉**：`ls -la` 查看隱藏檔或子目錄殘留

### 60_Shared 整合案例（2026-04-29）

| 階段 | 動作 | 結果 |
|------|------|------|
| 發現 | 讀到 `_misty-model.md` 實內容 | 認知盲區：誤判「用不上」 |
| 比較 | `wc -l` 比較 SKILL.md | 發現 llm-wiki-research 版本更完整 |
| 合併 | 實戰陷阱寫入現有 skill | 避免重複，保留最完整內容 |
| 驗證 | 每階段 `find` 確認 | 發現 index/schema 檔被遺漏 |
| 清理 | `rm -rf` 刪除子目錄 | `rmdir` 失敗後用 `ls -la` 找到殘留 |

## WSL 工具選擇警告

在 WSL 環境下，`terminal` 工具可能因 `$PWD` 損壞而完全失效（每次命令都報 `FileNotFoundError` 指向錯誤路徑）。遇到這種情況，**立即切換到 `execute_code`（Python `os` 模組）**：

```python
import os
# 列出目錄
items = sorted(os.listdir("/mnt/c/Users/安泰/OneDrive/Obsidian/..."))
# 讀取檔案
with open("/mnt/c/Users/安泰/OneDrive/Obsidian/.../file.md", encoding='utf-8') as f:
    content = f.read()
```

`execute_code` 不依賴 shell 工作目錄，永遠從 absolute path 讀取。

## 常見陷阱

| 陷阱 | 為什麼有問題 | 預防 |
|------|------------|------|
| 只用 `ls` 就判斷 | 檔名不等於內容用途 | 至少讀一個實質檔 |
| 假設我不用就是沒價值 | Misty 可能有用，只是我沒用到 | 問用戶確認 |
| 跳過 index.md | 結構定義是理解內容的第一步 | 永遠先讀 index |
| 忽略對話歷史 | 過去討論可能已有結論 | 先 `session_search` |
| 搬遷後不驗證 | 結構定義檔被遺漏在原位置 | 每階段 `find` 確認只剩 index/schema |
| 假設搬完了就完了 | 刪除失敗時沒有回頭檢查 | `ls -la` 確認後再 `rmdir` |
| 只看檔名決定去留 | `wc -l` 才能確認哪個版本更完整 | 比較後再決定合併或覆寫 |

## 蒸餾決策捷徑（用戶已明確意圖時）

當用戶說「全部蒸餾完砍」之類的明確指令時，**跳過回報確認，直接執行**。但仍須遵守：

1. **預先掃描一遍所有檔案**（用 `execute_code` + `os.listdir()` 快速分類）
2. **區分價值的層次**（自己原創內容 > 重新組織他人內容 > 純理論抄襲 > MOC 導航頁）
3. **預先計算「砍」的比例**，蒸餾完再統一報告
4. **不要每次問「這個要蒸餾嗎」**——浪費時間，用戶已經說完了

## 批次刪除捷徑（純備份檔）

常見 pattern：來源 vault 中有大量 `-110YK-7-109` 結尾的備份檔，與原檔 **完全相同（0 bytes 差異）**。

發現方式：
```python
# 比較原檔和備份檔大小
for orig, backup in pairs:
    orig_size = os.path.getsize(orig_path)
    backup_size = os.path.getsize(backup_path)
    if orig_size == backup_size:
        print(f"Identical backup (delete): {backup}")
```

處理方式：**直接刪除所有備份檔**，不需要讀內容確認。

## 蒸餾價值分類（評估時的判斷標準）

| 類型 | 說明 | 處理方式 |
|------|------|---------|
| **自己原創** | Misty 親自寫的個人化詮釋、案例 | 蒸餾進 atomic page |
| **重新組織** | 對他人內容的摘要、重組 | 蒸餾進 atomic page |
| **純理論抄襲** | 從書本/課程直接抄來，無個人化內容 | 檢查 wiki 是否已有；已有則跳過蒸餾 |
| **MOC 導航頁** | 純連結、無實質內容 | 直接刪除 |
| **整合失敗總結** | 嘗試過整合但放棄的紀錄 | 直接刪除 |
| **備份檔** | `-110YK-7-109` 或類似命名 | 直接刪除 |
| **空/極短檔** | < 100 bytes，只有 tagline | 直接刪除 |
