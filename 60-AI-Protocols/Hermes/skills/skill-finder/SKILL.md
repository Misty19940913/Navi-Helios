# Skill Finder — 發現「第一個最值得建的 Skill」

## 這個 Skill 做什麼

透過**結構化三題訪談**，找出「重複頻率最高、自動化回報最大」的那個工作缺口，然後給出具體的 Skill 建構建議。

不是教你寫 SKILL.md 語法——是幫你回答：「**我應該先建哪一個？**」

---

## 與 Claude Code skill-creator 的差別

> ⚠️ 這裡不是用 Anthropic 的 `skill-creator` CLI 工具（那是 Claude Code 專屬）。
> 這裡是將「**訪談發現法**」翻譯成 Hermes 可執行的流程。

---

## Step 1 — 三題訪談（強制，不可跳過）

用 `clarify` 工具一次發送三題：

### 問題一：重複類型

```
這個請求你最常在 AI 對話裡重複打的一種是什麼？挑一類就好。
```

| 選項 | 說明 |
|------|------|
| 寫作類 | 每次都要打「幫我寫 XX 風格的貼文/文章」，然後手動帶語氣範例 |
| 整理類 | 每次都要把一堆資料（文件、逐字稿、訊息）丟給 AI 整理成某個固定格式 |
| 回覆類 | 每次都要回某類訊息（客服、合作邀約），而且回的結構差不多 |
| 檢查類 | 每次都要跑一份檢查清單（發文前、上架前、會議前） |
| 查詢/決策類 | 每次都要查某種資料再做判斷（工具評估、方案比較） |
| 其他 | 以上都不完全對應，跟 AI 討論後再決定 |

---

### 問題二：觸發頻率

```
這類請求你多久會遇到一次？
```

| 選項 | 說明 |
|------|------|
| 每天 1 次以上 | 超高頻，建 Skill 回報率最高 |
| 每週 2-5 次 | 高頻，很值得建 |
| 每週 1 次 | 中頻，值得建但不急 |
| 每月幾次 | 低頻，可能先用筆記或範本就夠 |

---

### 問題三：輸出形式

```
你希望這個 Skill 做完後交付什麼？
```

| 選項 | 說明 |
|------|------|
| 一份 Markdown 文件 | 寫進某個資料夾讓你之後用 |
| 一段可以直接複製的文字 | 貼到社群、Email、文件裡 |
| 一個 JSON / 結構化資料 | 之後要丟到其他工具處理 |
| 一個帶決策的分析報告 | AI 幫你判斷，然後你再決定怎麼辦 |
| 其他 | 還不確定，先討論 |

---

## Step 2 — 根據三題組合，產出建議

組合三個答案，給出：

### 🥇 最推薦（1 個）

- **名稱建議**：`/[skill-name]`
- **做什麼**：具體描述流程
- **為什麼適合當第一個**：結合頻率 + 輸出形式 + 你的情境
- **預估耗時**：建這個 Skill 大概要多久

### 🥈 備選一（1 個）

同格式，但說明「如果第一個用了一週覺得不夠，再試這個」

### 🥉 備選二（1 個）

同格式

---

## Step 3 — 如果用戶答「其他 / 還在想」

啟動**反向挖掘模式**：

用 `clarify` 問：

```
問題 A：你昨天跟 AI 對話時，有沒有哪一段讓你覺得「我好像每次都在打一樣的東西」？
問題 B：上週有沒有哪個任務你覺得「下次不想再從零開始」？
```

根據答案回到 Step 2 邏輯。

---

## Step 4 — 確認迭代文化（觸發前必說）

在建議結尾加入這段：

> **迭代比一次到位更重要。**
> 
> 建完第一個 Skill 之後：
> - **用 1 週**再決定下一個建什麼
> - **如果某個 Skill 一週內沒觸發一次**——砍掉或改 description
> - **每次用完如果效果不夠好**——回來說「幫我改 /[skill-name]，讓它 XX」，AI 會直接改 SKILL.md

---

## Step 5 — 寫入候選追蹤清單

將結果寫入 `60-AI-Protocols/Hermes/skills/skill-candidates.md`（若無此檔則建立）：

```markdown
# Skill 候選追蹤清單

**已建立**：
- /[skill-name] — [一句話描述]

**建議接下來建（按優先度）**：
1. /[候補1] — [做什麼 / 為什麼適合] / 預估耗時
2. /[候補2] — [做什麼 / 為什麼適合] / 預估耗時

**規則**：
- 每週建 1 個，不要貪快
- 用 1 週後再決定值不值得建下一個
- 如果某個 Skill 一週內沒觸發，砍掉或改 description
```

---

## Step 6 — 銜接到建構流程

若用戶確認了方向，問：

> 要現在開始建 `/[skill-name]` 嗎？
> 
> 如果是，我會進入 `skill-authoring` 流程——幫你從意圖確認走到第一個可用的 SKILL.md。

## 與 skill-authoring 的分工

| 階段 | 工具 |
|------|------|
| **發現**：哪個最值得建？ | `skill-finder`（這個 Skill）|
| **建構**：怎麼寫 SKILL.md？ | `skill-authoring` |
| **驗證**：建完真的會被觸發嗎？ | `skill-authoring` 裡的觸發檢查 |
