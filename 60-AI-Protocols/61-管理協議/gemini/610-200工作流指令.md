## 一、 指令設計框架

所有自定義指令建議遵循以下結構，以便 Gemini 穩定解析：
1. **指令名稱 (Command Name)**：簡單易記的觸發詞（如 `//TASK_NEW`）。
2. **適用情境 (Context)**：在哪個資料夾或哪種對話下使用。
3. **執行邏輯 (Logic)**：AI 的思考步驟（先檢查什麼、後建立什麼）。
4. **輸出範本 (Template)**：最後產出的 YAML 與內文格式。

## 二、 自動化指令

### //sync
**功能**：重新掃描並載入 Navi Helios 3.5 系統的核心配置文件，確保當前對話上下文（Context）與最新的筆記規範、自動化指令保持 100% 同步。
- **觸發條件**：修改了管理協議後。 
- **後台動作**：重新讀取管理協議。 
- **Gemini 輸出規範**：回報「系統已更新」。

### //log
**功能**：將與AI之對話紀錄儲存，以利後續追蹤及分析。
- **觸發詞**：`//log` 
- **核心目標**：將當次對話的「知識增量」與「系統演進」進行結構化存檔。
- 輸出結構：[[AI對話日誌]]

### //rb
**指令格式**：//rb @[模板檔名] @[來源路徑/筆記檔名]
**功能**：將現有混亂筆記內容，套用系統標準模板進行重構。
**執行 SOP**：
1. **路徑解析**：
	- 第一個 `@` 後方為 **目標模板**（預設檢索 `600支援/620Template/`）。
	- 第二個 `@` 後方為 **待處理筆記**。
2. **內容調取**：
	- 若 Gemini 具備讀取檔案權限，將自動讀取兩者內容。
	- 若無法直接讀取，Gemini 應立即回覆：「請貼上 @[模板] 與 @[筆記] 的內容以便重構。」
3. **重構邏輯**：
	- **結構遷移**：將來源筆記的資訊，填入模板定義的欄位中。
	- **Metadata 校準**：檢查 YAML 是否符合 `610-100YAML規範`。
	- **術語優化**：根據你的知識風格進行潤稿。
4. **潤稿優化**：以專業及簡潔的語氣進行優化。

### //rv-day
- **功能**：
- **觸發條件**：每日覆盤或使用者手動輸入。 
- **後台動作**：執行 `scripts/daily_manager.ps1`。 
- **Gemini 輸出規範**： 
	- 摘要遷移了多少個任務。 
	- 列出遷移後的任務名稱與新日期。 
	- 詢問使用者是否要針對這些延期任務進行「殘酷裁決 (Kill List)」。

### //squeeze
- **功能**：提取靈感
- **適用路徑**：`000 捕獲 Inbox` 
- **執行邏輯**： 
	1. 提取靈感碎片中的「情緒核心」。 
	2. 調用 **610-440 (具體場景)**：將碎片轉化為一句具備「成像感」的核心主張。 
	3. 輸出：100字內金句 + 推薦應用情境。

### //break
- **功能**：願景拆解
- **適用路徑**：`100夢想/` 
- **執行邏輯**： 
1. 將感性描述轉化為理性架構。
2. 輸出三層級：使命 (Why) -> 任務 (What) -> 子任務 (How)。 
3. 自動生成符合 `230任務池/` 的任務 YAML 並預設 `status: todo`。

### //status
- **功能**：專案狀態判斷
- **適用路徑**：`300專案/` 
- **執行邏輯**： 
	1. 讀取 Readme 或進度紀錄。 
	2. 產出【已完成 / 進行中 / 待啟動】摘要。 
	3. 依據內容自動建議 3 條「下一步行動 (Next Action)」。

### //media
- **功能**：自媒體製作
- **適用路徑**：`500自媒體創作區/` 
- **執行邏輯**： 
	1. 同步啟動 **610-420 (開場)** 與 **610-450 (留白)**。 
	2. 輸出：IG 文案、三句互動金句、15秒短影片腳本（含運鏡提示）。

## //audit
- **目標**：在不進行重寫的前提下，以「冷峻主編」的視角對文本進行全方位的壓力測試。
- **執行邏輯**： 
	- **全域禁令掃描**：首先檢查文案中是否出現「禁忌語」（如：希望、讓我們...）或「結論式結尾」。
	- **四維度量化評分**：依照 `610-201` 定義的指標，給予 0-10 分的客觀評價。
	- **邏輯缺陷診斷**：指出內容在哪裡向平庸妥協、在哪裡顯得軟弱。
- **輸出規範**：
	- 報表應以 `[[自媒體文章稽核報告]]` 開頭，並使用三級標題 `### 自媒體文章稽核報告`。
	- 報表內容必須**排除** YAML 元數據區塊。
	- 每次執行 `//audit`，必須將此報告輸入該篇文章的最下方。

## //tweak
- **目標**：根據審計發現的缺陷，調用 [[610-010個人核心形象清單]] 作為全域約束，進行文字的「極致化處理」，並在尋找素材的多維解讀、減少語氣飄移的情況下，產出3篇新文章。
- #### 執行程序 (SOP)
1. **病灶讀取**：提取前一步 `//audit` 中評分低於 7 分的項目，作為本次手術的重點（例如：若脫水度低，則執行強效脫水）。
2. **五大模組對位**：
    - **對準 Hook**：若開場不夠冷峻，重新建立身份衝突。
    - **對準視角**：將任何「請求認同」的語氣轉化為「強者選擇」。
3. **場景壓制**：強化文字中的物理細節（黑膠、冷氣聲、筆錄室的燈光），增加氛圍的重量。
4. **全域禁令複檢**：確保重構後的結尾沒有結論式廢話，且完全移除禁忌語。

### //meta_fix
- **功能**：生成模板使用說明與應用場景
- **適用路徑**：`620Template/`
- **執行邏輯**：分析 Markdown 模板結構，生成使用說明與應用場景。

### 1. 原子任務捕獲 (//CAPTURE_TASK)

**用於快速將對話內容轉化為 230 任務池的筆記。**
- **觸發條件**：當我說「幫我記下這個任務」或輸入 `//CAPTURE_TASK`。
- **執行步驟**：
    1. 提取任務名稱，格式化為 `任務描述`。
    2. 詢問 `subject` (範疇) 與 `parent` (連結專案)。
    3. 自動生成符合 `610-100YAML規範` 的 7 行標準 YAML。
    4. 預設 `status: draft`，`scheduled: today`。
- **存儲位置**：`200 週期紀錄/230 任務池/`。
    

### 2. 自媒體創作啟動 (//CONTENT_START)

**用於從零開始構思 500 區的內容。**

- **觸發條件**：輸入 `//CONTENT_START` 並提供主題。
    
- **執行步驟**：
    
    1. 建立 `#系統/創作` 標籤筆記。
        
    2. 插入 `quadrant: "職業"`。
        
    3. 自動鑲嵌 `分享::`、`按讚::`、`留言::`、`轉發::` 於筆記末尾。
        
    4. 提供三個標題建議與內容大綱框架。
        

### 3. 失敗裁決審計 (//FAIL_AUDIT)

**用於清算已過期但未完成的任務。**

- **執行邏輯**：
    1. 掃描所有 `status: archived` 但 `fail-reason` 為空的筆記。
    2. 逐一詢問我：「為何這個任務失敗了？」
    3. 將我的回答填入 YAML 的 `fail-reason` 欄位。

## 三、 如何讓 Gemini 執行這些指令？

你可以在 `GEMINI.md` 的末尾加入這行權力聲明：

> **「當我輸入以 `//` 開頭的指令時，請優先檢索 `600 支援系統/630 工作流指令集.md` 中的邏輯進行回應。」**

## 四、 建議開發清單

既然你要自定義，以下是我認為對你目前的架構最有幫助的幾個指令，你可以優先設計：

- `//MOC_SYNC`：自動檢查並更新 MOC 中的 Dataview 代碼。
    
- `//JOURNAL_REFLECT`：在日誌中自動根據當日完成的任務，生成一段自我覺醒（心智象限）的總結。
    
- `//METADATA_FIX`：批量修正不符合 610 規範的舊筆記屬性。
