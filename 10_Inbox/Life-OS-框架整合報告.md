| ----------------------- | ----------------------------------------------------------------------- | ------------------------------------------------------ |

### 1.2 筆記數量與標籤規則（朱麒）

- 每一篇筆記只標記**一種**標籤
- 四種標籤的學習流程對應：

```
了解名詞 → 閱讀資料 → 自己思考 → 歸類主題
(#📖)      (#📥)       (#🔗)       (#🗺)
名詞筆記   來源筆記     關係筆記     主題筆記(MOC)
```

### 1.3 PARA 結構（LYT / quanru）

```
├── 0. PeriodicNotes  (日/週/月/年週期性筆記)
├── 1. Projects       (主動執行中的專案)
├── 2. Areas          (長期責任領域，不需完成但需維護)
├── 3. Resources      (感興趣但暫無行動的主題)
├── 4. Archives       (擱置的舊專案/資源)
└── 5. Express        (表達/輸出的創作)
```

> **Areas vs Projects**：Areas 沒有終點（如「健康」），Projects 有（如「減重 5 公斤」）。

### 2.2 HQ 處理中心內部子系統

#### PKM 知識系統（PMM/PIM/PWM/PSM/PRM 六子系統）

PKM 不是單一系統，而是六個領域的混合色輪（Nick Milo）：

| 代號          | 名稱                      | 核心功能               | 典型行為           |
| ------------- | ------------------------- | ---------------------- | ------------------ |
| **PMM** | 記憶管理 (Memory)         | 收集知識、發展記憶     | 收藏文章、讀書筆記 |
| **PIM** | 想法管理 (Idea)           | sense-making、連結想法 | 建立關係筆記、MOC  |
| **PWM** | 寫作管理 (Writing)        | 包裝與分享內容         | 寫文章、發布創作   |
| **PPM** | 生產力管理 (Productivity) | 推進專案與目標         | 執行待辦、追蹤進度 |
| **PSM** | 技能管理 (Skill)          | 發展能力與習慣         | 練習、間隔重複     |
| **PRM** | 關係管理 (Relationship)   | 維護人脈與溝通         | 紀錄來往、追蹤關係 |

> 這六個領域**經常重疊**：連接想法時在寫作論述，寫作時在推進專案。PKM Planet 是混合色輪，不是線性流程。

#### PM 專案管理 — Efforts 取代 Projects

**為什麼「努力(Efforts)」比「專案(Projects)」更真實？**

| Projects（傳統）   | Efforts（解放）      |
| ------------------ | -------------------- |
| 終點導向（交付物） | 過程導向（持續生長） |
| 有明確交付物       | 更真實反映思考狀態   |
| 壓迫感強           | 自由感               |
| 完成後封存         | 完成後可持續更新     |

**四種強度（Four Intensities）：**

| 強度              | 狀態   | 意義   | 適用情境                       |
| ----------------- | ------ | ------ | ------------------------------ |
| 🔥`On`          | 進行中 | 最活躍 | 正在燃燒的優先事項             |
| ♻️`Ongoing`   | 持續中 | 更廣泛 | 長期興趣、持續學習             |
| 〰️`Simmering` | 沉澱中 | 醞釀   | 暫時擱置的想法，等待更好的時機 |
| 💤`Sleeping`    | 沉睡中 | 冷儲存 | 已完成、隨機、長期封存         |

> **旋轉向下**：`On` 太多？將一項移到 `Simmering`，立刻感受到頻寬釋放。

#### Calendar 行事曆系統

- **Logs**（日誌）：每日/每週/每月流水筆記
- **Notes**（筆記）：
  - `Compass`：計劃與回顧
  - `Past Years`：過往年度歸檔

#### MOC 統一入口

MOC 是 HQ 的指揮中心，連接 PKM/PM/Calendar 所有子系統。

## 🔷 三、MOC（內容地圖）—— 跨框架共識樞紐

### 3.1 MOC 定義

**MOC = Map of Content（內容地圖）**，發音為 "Em-Oh-Cee"。

MOC 是一個**資訊叢集**，將「事物」與其他「事物」在上下文中進行映射。它是覆蓋層（Overlay），而非資料夾的剛性結構——刪除 MOC 不會刪除任何基層筆記。

### 3.2 九個使用 MOC 的理由（Nick Milo）

1. **管理焦慮**：提供專用空間匯聚相關筆記，積極降低心理負荷
2. **加速 ideation**：在濃縮環境中鼓勵想法快速碰撞與交流
3. **參考工具**：完成一個 MOC 後，專案已完成 80%（如文章、產品、總結）
4. **導航工具**：連結 vault 其餘部分的可靠入口
5. **提醒工具**：多年後返回，快速記起某主題的完整脈絡
6. **持續演化**：跟隨你的想法成長、成熟、增加複雜性
7. **保持流動**：非破壞性、非限制性，不像資料夾那樣逼迫你用它來存取筆記
8. **無限彈性**：不像資料夾或目錄（TOC），MOC 是**異層級（Heterarchy）**的——不同 MOC 可以用不同方式映射相同資訊
9. **空間記憶**：想法作為星座的一部分被記住，而非孤立漂浮（古希臘 Method of Loci）

### 3.3 MOC 三階段（Nick Milo）

何時使用 MOC：當你感受到** Mental Squeeze Point（心智壓力點）** 時——那就是要製作新 MOC 的信號。

```
Gather (收集) ──→ Collide (碰撞) ──→ Navigate (導航)
  ↓                  ↓                  ↓
 匯聚碎片         強制創意連結        結構化呈現
```

| 階段                    | 目的                                   | 心理狀態     |
| ----------------------- | -------------------------------------- | ------------ |
| **Gather 收集**   | 將散落的相關筆記匯聚到一個 MOC         | 焦慮 → 舒緩 |
| **Collide 碰撞**  | 強制讓想法互相作用，發現隱藏連結與洞見 | 探索 → 驚喜 |
| **Navigate 導航** | 結構化整理，成為可靠的參考與入口       | 複雜 → 清晰 |

### 3.4 Fluid Frameworks vs Rigid Frameworks

| 維度     | Fluid Framework（MOC/Home/原子筆記） | Rigid Framework（資料夾/目錄） |
| -------- | ------------------------------------ | ------------------------------ |
| 結構     | 非剛性、有機、可按需使用             | 剛性、必須、刪除即消失         |
| 刪除後果 | 筆記依然存在                         | 筆記路徑受影響                 |
| 強制性   | 非必要支援結構                       | 必要支援結構                   |
| 組織彈性 | 筆記可同時被多個 MOC 引用            | 筆記只能在一個資料夾中         |
| 典型工具 | MOC、Home note、高階原子筆記         | 作業系統資料夾                 |

> MOC 是「覆蓋層」——添加相關資訊，但不影響基層筆記。筆記是自由的，可以被任意數量的 MOC 引用。

### 3.5 Home Note 作為 PKM 入口

Home note 是 vault 的起點，類似網站首頁。它的作用：

- 擔任 MOC Library 的入口
- 透過「up」連結向上導航到更高層級
- 透過「down」連結進入具體內容

```
Home note
  ↓ (down)
Ideaverse Map
  ↓ (down)
On the process of note-making
  ↓ (up)
Ideaverse Map
  ↓ (up)
Home note
```

> 這就是 Nick Milo 說的「星圖」——在不同想法層級之間進行 hyperspace jumps。

## 🔷 四、防彈筆記法 — 深度解析（框架 C / Esor Huang）

### 4.1 核心範式轉移

> **筆記不是倉庫，是行動的副產品。**

| 維度     | 傳統靜態分類             | 防彈筆記（代謝系統） |
| -------- | ------------------------ | -------------------- |
| 核心目標 | 資訊的分類、保存         | 推動任務進度與產出   |
| 筆記單位 | 來源或時間               | 任務/目標            |
| 演化狀態 | 靜態，寫完後封存         | 動態，隨執行持續修改 |
| 組織邏輯 | 分類導向                 | 任務導向             |
| 更新頻率 | 極低                     | 極高                 |
| 失敗模式 | 收藏家謬誤（儲存≠掌握） | 筆記與行動脫節       |

### 4.2 筆記五階段生命週期

```
Inbox (收集箱) → Temp (暫時筆記) → Project (專案任務) → Resource (資源區) → Evergreen (永久筆記)
   ↓                ↓                  ↓                   ↓                  ↓
 緩衝區          初步轉化           行動指揮            長期參考            智慧結晶
```

| 階段                | 角色                   | 演化原則                             |
| ------------------- | ---------------------- | ------------------------------------ |
| **Inbox**     | 接收一切，零門檻       | 僅做初步捕獲                         |
| **Temp**      | 等待處理的暫時地帶     | 若 48 小時內無行動，評估是否值得保留 |
| **Project**   | 單一任務的行動指揮中心 | 任務結束後移出或蒸餾至 Evergreen     |
| **Resource**  | 長期參考資源庫         | 定期檢視是否需要更新                 |
| **Evergreen** | 活得夠久的智慧結晶     | 持續維護，隨個人成長而演化           |

### 4.3 PAA 結構

每則筆記承擔三種角色之一：

- **P (Project)** — 任務筆記，單一任務一則筆記
- **A (Action)** — 行動流程，按執行步驟排序（下一步行動清單）
- **A (Archive)** — 封存，任務結束後移出視線

### 4.4 3問3動處理框架（每則筆記的命運判定）

**釐清現況（3問）：**

| 問題    | 意圖      | 回答範例                       |
| ------- | --------- | ------------------------------ |
| Why/Who | 目的/對象 | 這筆記是為了什麼專案？給誰看？ |
| How     | 量化指標  | 如何知道這筆記「完成」了？     |
| What    | 阻礙限制  | 什麼阻礙我現在就行動？         |

**決定未來（3動）：**

| 行動           | 做法                                   |
| -------------- | -------------------------------------- |
| **改寫** | 用自己的語言重新表述（強制的理解檢驗） |
| **連結** | 建立與現有知識網的參照（MOC/相關筆記） |
| **提取** | 轉化為具體待辦事項（下一步行動）       |

### 4.5 核心指令

> **更新勝於新增。**
> 高效防彈系統：每日新增 2-3 則，每日更新 30+ 則舊筆記

> 拒絕收藏家謬誤：儲存資訊不等於掌握資訊。

## 🔷 五、Graph View 寫作流程（框架 D / 朱麒）

### 5.1 四種筆記類型

| 類型     | 標籤 | 說明                             | 範例                                     |
| -------- | ---- | -------------------------------- | ---------------------------------------- |
| 名詞筆記 | #📖  | 專有名詞解釋                     | `#📖 沉沒成本`：已付出且不可回收的成本 |
| 來源筆記 | #📥  | 外界知識（讀書/Podcast/文章）    | `#📥《原子習慣》筆記`                  |
| 關係筆記 | #🔗  | 自己思考後寫下的筆記             | `#🔗 習慣疊加的本質是情境提示`         |
| 主題筆記 | #🗺  | Map of Content，能囊括同主題內容 | `#🗺 習慣系統 MOC`                     |

> 每一篇筆記只標一種標籤。

### 5.2 Local Graph View 使用技巧

**1. 設定群組顏色**

- `...` > `Open local graph view` > `Groups` > `New group`
- 設定四色：來源📥=綠、關係🔗=粉紅、名詞📖=土黃、主題🗺=藍

**2. 輸入篩選語法**

- `tag:#📥` → 只看某類筆記

**3. 調整 Filter Depth**

- Depth=1：只看直接連結
- Depth=2：再多顯示一層
- 依此類推到 Depth=5

### 5.3 Graph View 大量寫作五步驟

> **核心論點：寫作卡關是因為參考資料不夠。當我們不必從一張白紙開始時，創作的過程快上許多。**

**步驟 1：設定界面**

- 左側開啟「名詞筆記」
- 右側開啟 Graph View（按住 Cmd 再點擊 Graph View 按鈕，變成左右各半）

**步驟 2：在 Graph View 搜尋關鍵字**

- 使用 `line:(關鍵字)` 語法
- 關聯圖上只顯示含有該關鍵字的筆記節點

**步驟 3：雙視圖對照**

- 在左側筆記中使用 `line:(關鍵字)` 的 Obsidian Query
- 顯示含有該關鍵字的段落文字，與右側 Graph View 同步對照

**步驟 4：蒐集碎片**

- 將有感觸的內容轉貼到另一則「創作筆記」
- 收集碎片化靈感

**步驟 5：主動建立連結**

- 在 Graph View 中發現「有提到但沒有實際連結」的筆記時
- 主動點擊建立 wikilink

### 5.4 寫作流程

```
碎片化文字 → 整理成完整段落 → 列下文章大綱 → 把段落依序填入大綱
```

## 🔷 六、PARA + ACE 資料夾框架

### 6.1 PARA（來源：A — quanru / LYT）

```
├── 0. PeriodicNotes  (日/週/月/年筆記)
├── 1. Projects       (有明確交付物的專案)
├── 2. Areas          (長期責任領域，無終點)
├── 3. Resources      (感興趣但暫無行動)
├── 4. Archives       (已完成/擱置)
└── 5. Express        (表達/輸出)
```

### 6.2 ACE 思維空間框架（來源：B — Nick Milo / Ideaverse）

> **只要 3 個資料夾，擁有一生使用的資料夾結構。**

ACE 是三個**思維空間**：

| 思維空間           | 定向鏡頭 | 意圖 | 組織原則      | 潛在好處    | 引導問題         |
| ------------------ | -------- | ---- | ------------- | ----------- | ---------------- |
| **Atlas**    | 知識     | 理解 | 相關性 / 空間 | 學習 / 記憶 | *你想去哪裡？* |
| **Calendar** | 時間     | 專注 | 時間          | 記憶 / 反思 | *你在想什麼？* |
| **Efforts**  | 行動     | 行動 | 重要性        | 創造        | *你能做什麼？* |

**各空間內容定位：**

| 思維空間           | 主要內容                                                                                                |
| ------------------ | ------------------------------------------------------------------------------------------------------- |
| **Atlas**    | 地圖（Maps）、原子筆記（Dots）、來源（Sources）、想法（Ideas）、輸出（Outputs）、人脈（People）、Vaults |
| **Calendar** | 日誌（Logs）、每日/每週/每月筆記、指南針（Compass，回顧與計劃）、過往年度（Past Years）                 |
| **Efforts**  | On（最活躍）、Ongoing（持續）、Simmering（沉澱）、Sleeping（沉睡）                                      |

**初學者 → 導航者 → 禪師的演化：**

| 階段   | Atlas                                        | Calendar                           | Efforts                       |
| ------ | -------------------------------------------- | ---------------------------------- | ----------------------------- |
| 初學者 | 平板結構                                     | 平板結構                           | 平板結構                      |
| 導航者 | Maps / Notes（Ideas/Sources/Outputs/People） | Logs / Notes（Compass/Past Years） | On/Ongoing/Simmering/Sleeping |
| 禪師   | 回到平板                                     | 回到平板                           | 回到平板                      |

> 禪師階段不是退化，而是從複雜回歸簡約——但帶著導航者積累的所有智慧。

### 6.3 ACE 的鐘擺效應

> *生活是漫長的，鐘擺會擺動。*

你可能在：

- **Efforts**（行動）中 → 發現新興趣 → 轉移到 **Atlas**（知識）
- **Atlas**（知識）中 → 新工作需要快速記錄 → 轉移到 **Calendar**（時間）
- **Calendar**（時間）中 → 承諾感到穩定 → 轉移回 **Efforts**（行動）

### 6.4 附助資料夾：+ 和 x

| 資料夾      | 用途                                         |
| ----------- | -------------------------------------------- |
| **+** | 任何不符合 Atlas/Calendar/Efforts 的額外內容 |
| **x** | 實驗性、過渡性、不確定的內容                 |

## 🔷 八、常青筆記（Evergreen Notes）—— 活得夠久的智慧

### 8.1 筆記者 vs 做筆記的人

|          | 筆記者 (Note-takers)  | 做筆記的人 (Note-makers)      |
| -------- | --------------------- | ----------------------------- |
| 目的     | 消耗資訊（考試/專案） | 創造 living notes（持續生長） |
| 筆記壽命 | 里程碑結束後失去價值  | 活得夠久，價值持續累積        |
| 典型行為 | 收藏、抄錄            | 改寫、連結、演化              |
| 長期結果 | 噪音累積              | 智慧結晶                      |

### 8.2 常青筆記四大原則

1. **清晰且簡潔** — 標題是清晰陳述句，不是關鍵詞（例：「習慣形成的本質是情境提示的疊加」，而非「習慣」）
2. **用自己的話** — 強制深度理解，自然展現獨特視角
3. **已連結** — 用清晰的話語自然產生 wikilink
4. **非靜態** — 可隨新經驗重寫、重構，隨時間增值

> 不要強迫筆記「atomic」—— 遵循金髮姑娘原則：不太短，不太長。
> 如果筆記太長，你會知道——因為它試圖表達太多不同的事情。

### 8.3 常青筆記的長期複利

- **可組合性**：一個洞見筆記 + 另一個 → 意外可選項（unexpected optionality）
- **時間複利**：越多年，連結越多，洞察越深刻
- **反脆弱**：筆記本身經得起時間考驗

> 常青筆記是 Andy Matuschak 提出並由 Nick Milo 在 Obsidian 社群發揚的概念，源自 Niklas Luhmann 的 Zettelkasten。

## 🔷 十、實踐建議

### 10.1 入門路徑（四週循環）

| 週次             | 重心                | 具體行動                                            |
| ---------------- | ------------------- | --------------------------------------------------- |
| **第一週** | 建立入口 + 理解框架 | 建立 Inbox 收集系統，理解防彈筆記的 3問3動          |
| **第二週** | 組織結構            | 導入 PARA 或 ACE 結構，設定 Home Dashboard          |
| **第三週** | MOC 實踐            | 練習 MOC 三階段（收集→碰撞→導航），從單一主題開始 |
| **第四週** | 寫作產出            | 使用 Graph View 進行一次完整寫作（步驟1-5）         |

### 10.2 進階原則

- **整理發生在執行時**（Organize while doing）
- **拒絕收藏家謬誤**：儲存 ≠ 掌握
- **更新勝於新增**：修改 30+ 筆記 > 新增 3 筆記
- **MOC 是入口**：讓所有資訊最終匯聚至主題地圖
- **發散與收斂平衡**：白板自由探索 ↔ 資料庫結構收斂
- **Efforts 的四種強度**：時常檢視，動態調整頻寬

### 10.3 常見陷阱

| 陷阱                       | 對應                                       |
| -------------------------- | ------------------------------------------ |
| 過早設計完美結構           | 先用預設 ACE，結構需被證明合理性後才自訂   |
| 把收斂工具用於發散階段     | 白板還沒收斂前不要放進資料庫               |
| 只新增不更新               | 每篇新筆記代價 = 更新 10 篇舊筆記          |
| MOC 太早固化               | Gather → Collide → Navigate 順序不能跳過 |
| 笔记者心态（只记录不创作） | 設定輸出目標：用 Graph View 每週寫一篇     |
