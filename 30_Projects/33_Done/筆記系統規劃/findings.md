## 輸出驅動的核心原理

**傳統輸入驅動方式**
- 讀書/文章 → 筆記 → 存起來
- 問題：筆記與實際應用脫節
- 結果：筆記一堆但從來不用

**輸出驅動方式**
- 明確目標（輸出）→ 尋找所需（輸入）→ 筆記服務產出
- 每筆記都是「被使用」的零件
- 確保筆記有明確的服務對象

**價值**
| 面向 | 說明 |
|------|------|
| 效率 | 只收集需要的資訊 |
| 可用性 | 筆記直接可用 |
| 目的性 | 每筆記都有意義 |

## 輸入-輸出映射關係設計

**映射結構**
```
輸出（目標）→ 所需輸入（筆記需求）→ 筆記內容
```

**三層映射**
| 層級 | 輸出類型 | 需要的輸入 |
|------|----------|------------|
| 戰略層 | 人生目標、5年計畫 | 價值觀、願景、能力評估 |

| 執行層 | 文章、任務、決策 | 具體數據、案例、方法論 |

### 可新增的實務做法

1. **72小時入系統規則**
   - 任何筆記創建後72小時內必須分類、貼標籤、寫 description
   - 否則自動移入緩衝區或提醒處理

2. **雙向 description 設計**
   ```yaml
   description: |
     📥 來源：這則資訊解決什麼問題？
     🎯 輸出：指向前述三層中的哪一層？
     🔗 關聯：與哪些現有筆記連結？
   ```

3. **輸入類型標籤**（對應卡片盒四種筆記）
   - `#fleeting`（瞬時靈感）→ 24h內處理
   - `#project-bound`（項目專用）→ 項目結束後歸檔
   - `#permanent`（永久知識）→ 進入知識庫
   - `#strategic`（戰略層）→ 定期複審

4. **月度/季度複習流程**
   - 每月檢視：fleeting 筆記處理狀態
   - 季度檢視：戰略層筆記是否仍符合目標

## 二、漏洞分析

### 未覆蓋的關鍵環節

1. **無收藏者謬誤檢核機制**
   - 系統允許無限囤積，缺少「儲存 ≠ 學習」的驗證設計
   - 無法區分「暫存區」與「長期記憶區」的流轉規則

2. **無成熟度標籤體系**
   - 缺少「種子→芽苗→果實」的成長階段標記
   - 無法判斷筆記是否已可提取用於輸出

3. **PARA 與卡片盒未整合**
   - 兩套方法論並存但無橋接機制
   - 專案導向（PARA）與發現導向（卡片盒）的取捨邏輯缺失

4. **缺乏「為未來自我解釋連結理由」**
   - 回溯時無法理解為何建立該連結
   - 上下文蒸發後連結變成死鏈

### 生命週期管理缺口

1. **無 24-48h 處理原則**
   - 缺乏「擱置太久」的回收/昇華機制
   - 筆記無過期或重審提示

2. **無原子化檢核**
   - 筆記顆粒度標準缺失
   - 無法判斷資訊是否已分解至可組合狀態

3. **輸出節點無標記**
   - 哪些筆記已產出、哪些仍待轉化不可見
   - 知識→創作的轉換率無法追蹤

### 導航與發現機制不足

1. **依賴 tag 線性搜尋，缺乏 MOC 聚合**
   - 主題入口單一，無法瀏覽性探索
   - 多維度交叉檢索缺失

2. **無情境標記**
   - 無法區分「研究用」「創作用」「參考用」的差異
   - 同一筆記在不同產���境

3. **無知識湧現觸發機制**
   - 筆記間隱性關聯無法被主動揭示
   - 只能依賴人工主動發現

### 知識產出流程缺口

1. **輸出導向斷裂**
   - 輸入端無對齊明確產出目標
   - 蒐集與創作各自為政

2. **三層映射虛置**
   - 戰略/戰術/執行層映射缺乏落地路徑
   - 無法從戰略目標反向滲透至筆記蒐集

3. **心理擠壓點未納入**
   - 忽略「必須產出」的時間壓力如何影響筆記行為
   - 無催促進展的機制

## 三、自動化鋪陳分析

### NOMA 詳解（2026-04-15 新增）

NOMA 是 **Linking Your Thinking (LYT)** 系統的核心思考框架，用四個問題驅動筆記從「記錄」升級為「創作」。

#### 四個問題定義

| 縮寫 | 原文 | 中文翻譯 | 筆記應用 |
|------|------|----------|----------|
| **N** | **N**ote what this is | 這是什麼？ | 定義筆記的核心概念 |
| **O** | **O**utline why it matters | 為什麼重要？ | 說明這個資訊的價值 |
| **M** | **M**ap what I can do with it | 我能用它做什麼？ | 連結到行動/產出 |
| **A** | **A**ssociate how it connects to... | 與什麼相關聯？ | 建立筆記間的連結 |

#### 實務範例

**傳統筆記（被動記錄）**
```markdown
# 複利效應
複利是指利息滾入本金繼續產生利息。
公式：FV = PV × (1 + r)^n
```

**NOMA 筆記（主動創作）**
```markdown
# 複利效應

## N - 這是什麼？
複利是利息滾入本金繼續產生利息的增長模式。

## O - 為什麼重要？
微小差異在長期會產生巨大差距（e^rt 指數增長）。

## M - 我能用它做什麼？
- 個人財務：定期投入ETF
- 知識累積：每天學習一小時
- 習慣養成：微小改變×長期

## A - 與什麼相關聯？
- [[K-FIN-001-1_WACC公式與定義]]（財務應用）
- [[K-LIF-003-時間複利]]（習慣應用）
```

#### 核心價值

| 層面 | 傳統筆記 | NOMA 筆記 |
|------|----------|-----------|
| 目的 | 儲存資訊 | 驅動思考 |
| 狀態 | 靜態（資料庫） | 動態（思考引擎）|
| 使用 | 以後可能用到 | 馬上可提取 |
| 連結 | 無 | 有脈絡、可解釋 |

#### 與現有系統整合方式

在 description 區塊應用 NOMA：
```yaml
description: |
  📥 N：這是什麼？
     [核心定義]
  
  📥 O：為什麼重要？
     [價值說明]
  
  📥 M：我能用它做什麼？
     - [行動1]
     - [行動2]
  
  🔗 A：與哪些筆記相關？
     - [[筆記A]]
     - [[筆記B]]
```

#### NOMA 的自動化潛力

- **連結理由檢查**：在 A 階段自然寫出「為什麼連結」，解決未來回溯問題
- **驅動輸出**：M 階段直接對應「三層輸出映射」
- **知識湧現**：A 階段建立的關聯是 MOC 自動生成的基礎

---

### 可自動化的規則/流程

| 自動化項目 | 觸發條件 | 預期動作 | 優先度 |
|-----------|---------|---------|--------|
| 靈感筆記未處理提醒 | 筆記建立後 24~48小時且無 `#processed` | 發送提醒訊息 | 中 |
| 文獻/永久筆記分類 | 筆記包含 `source:` 欄位 | 自動標記 `#type:literature` 或 `#type:permanent` | 中 |
| 輸出標籤格式驗證 | 筆記 YAML 包含 `output:` 欄位 | 檢查格式是否符合規範 | 高 |
| 成熟度標籤自動賦予 | 筆記內容 < 200 字元 或無連結 | 自動標記 `#seed` | 中 |
| NOMA 標籤生成 | 筆記包含 `noma:` 欄位 | 生成對應標籤 | 低 |
| 相同標籤筆記彙整 | 同標籤筆記數 ≥ 3 | 建議建立或更新 MOC | 中 |
| 相關領域筆記碰撞 | 筆記包含多於 2 個不同領域標籤 | 建議可能相關的其他筆記 | 低 |
| 原子化檢查 | 筆記字數 > 800 或 > 3 個 ## 標題 | 警告可能不夠原子化 | 高 |
| 自律性檢查 | 筆記缺少核心概念定義 | 警告可能無法脫離脈絡理解 | 中 |
| 連結理由檢查 | 建立 `[[wikilink]]` 連結時 | 提示需补充連結理由 | 低 |
| PARA 行動力判斷 | 筆記內容包含 action/todo 等詞 | 建議移至 Projects 或 Knowledge | 中 |
| Projects vs Areas 區分 | 筆記描述有特定完成時間 | 建議 Projects；反之 Areas | 中 |
| 無感組織建議 | 閒置筆記 > 30天無存取 | 建議複習或移至 Archives | 低 |
| 湧現觸發提醒 | 同一領域筆記數 ≥ 5 且無 MOC | 建議建立該領域 MOC | 中 |

### 需要定義的配置

| 參數 | 建議值 | 說明 |
|------|-------|------|
| seed 字數閾值 | < 200 字 | 未成熟的筆記 |
| fruit 閾值 | ≥ 1 個 connections 或 related | 已成熟的筆記 |
| MOC 觸發閾值 | ≥ 3 篇相同標籤筆記 | 建議建立 MOC |
| 原子化閾值 | > 800 字或 > 3 個 ## 標題 | 建議拆分 |
| 閒置提醒天數 | 30 天 | 無感組織 |
| 靈感筆記提醒 | 24~48 小時 | 未處理提醒 |

| **輸出驅動輸入原則** | 系統存在的前提，防止無限膨脹 |
| **三層輸出映射** | 現有 vault 000-400 分層的理論基礎，已運作穩定 |
| **tags + description 搜尋** | 實際檢索機制，輕量且直觀 |
| **vault 分層結構** | 物理組織，實用且驗證有效 |

### 輔助（可用可不用）


| 元件 | 理由 |
|------|------|
| **72小時入系統規則** | 有助於輸入紀律，但非系統運作必要條件 |
| **四種筆記類型** | 可作為「視角篩選」而非強制分類 |
| **ARC 複習循環** | 長期知識活化有益，但初期可暫緩 |
| **MOC 彙整** | 手動式 MOC 已足夠，自動化的邊際效益存疑 |

### 過度設計（建議移除）


| 元件 | 理由 |
|------|------|
| **14項自動化規則** | 與「直接用 tags 搜尋」的輕量原則衝突，疊加過重 |
| **成熟度標籤** | 尚未驗證需要性，先引入會增加認知負擔 |
| **timing 標籤** | 同上，且與描述重疊 |
| **輸入類型標籤** | 與 description 功能重疊 |

## 五、回圈機制設計

### 評估點

| 評估維度 | 指標 | 頻率 |
|----------|------|------|
| **標籤使用效率** | 實際被搜尋的 tags 佔比 | 季度 |
| **筆記流動率** | 從「待處理」進到「可用」的比率 | 月度 |
| **三層映射命中率** | 從輸出反向找到輸入的成功率 | 月度 |
| **MOC 彙整價值** | 彙整後被引用次數 | 季度 |

### 反饋方式

- **被動反饋**：每次搜尋後記錄「找到/找不到」，累積作為調整依據
- **主動反饋**：每季檢視一次 vault 結構，識別閒置分層或冗餘 tags
- **失敗反饋**：當某個輸出映射找不到足夠輸入時，觸發「輸入缺口報告」

### 迭代週期

| 週期 | 動作 |
|------|------|
| **週** | 快速檢視：新增 tags 是否有意義 |
| **月** | 中期檢視：哪些標籤/分層未被使用 |
| **季** | 系統檢視：輸出驅動原則是否仍適用，架構是否需要重構 |
| **年** | 大幅檢視：是否需要版本升級（如 v0.2.0 → v0.3.0） |

**刪除原則**：當某個功能連續兩季未被觸發或無明顯效益果斷移除。

## 總結

v0.2.0 的「輸出驅動輸入」是正確的核心方向。新增的 7 類建議中，**真正值得引進的只有 3 項**：72小時規則、手動 MOC、季度 ARC。其餘建議大多與現有機制重疊或為時尚早。

**核心原則**：保持輕量，寧可少做也不要無腦疊加。每增一項功能，必須問「這解決了什麼現有痛點？」而非「別人都有所以我也要」。
