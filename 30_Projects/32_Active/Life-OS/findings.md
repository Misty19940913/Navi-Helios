## 一、LYT / Ideaverse 研究啟發與系統演化 (2026-05-02)

### 1.1 為什麼要從 v3.0 轉向 v4.0？

經過對 `LYT Kit` 與 `Ideaverse Lite 1.5` 的深度審計（詳見 [[LYT_Analysis]]），我們發現現有系統存在以下「結構性摩擦」，這是推動本次計畫改變的核心原因：

- **複雜度過載**：原有的 ACCESS 系統或自定義的八大支柱對大腦的即時導航仍有壓力。
- **專案管理僵化**：傳統的「進行中/已完成」狀態無法反映創意工作的「強度」差異，導致心理壓力熵增。
- **知識孤島**：`40_Knowledge` 缺乏一個具備「全球導航」能力的通用分類索引，導致筆記存放位置高度依賴個人直覺。

### 1.2 核心發現與改進方向

| 研究對象 | 核心發現 (Insights) | Life-OS 2.0 (v4.0) 採納方案 |
| :--- | :--- | :--- |
| **KCE 框架** | 將系統精簡為 Knowledge, Calendar, Efforts。 | **結構重組**：將現有資料夾與 MOC 重新對標至 KCE 三大模塊。 |
| **Effort 強度** | 專案不應只有狀態，應有「火熱程度」標籤。 | **強度標籤**：引入 `On / Ongoing / Simmering` 取代單一狀態。 |
| **Library 索引** | 000-900 分類法是知識導航的萬用標準。 | **導航入口**：建立基於杜威分類法的 `Library MOC`。 |
| **觀點型筆記** | 標題應是一個陳述句 (Statement)，推動思考。 | **命名規範**：在 `10_Cards` 推行觀點型標題規範。 |
| **冷卻墊機制** | Inbox 需要緩衝期以過濾無效資訊。 | **捕捉 SOP**：在 `10_Inbox` 建立冷卻與週期性清理機制。 |


### 5.3 PARA 結構（LYT / quanru）

```
├── 0. PeriodicNotes  (日/週/月/年週期性筆記)
├── 1. Projects       (主動執行中的專案)
├── 2. Areas          (長期責任領域，不需完成但需維護)
```

## 七、MOC 深度解析（跨框架共識樞紐）

### 7.1 MOC 定義

**MOC = Map of Content（內容地圖）**。

MOC 是一個**資訊叢集**，將「事物」與其他「事物」在上下文中進行映射。它是覆蓋層（Overlay），而非資料夾的剛性結構——刪除 MOC 不會刪除任何基層筆記。

### 7.2 九個使用 MOC 的理由（Nick Milo）

1. **管理焦慮**：提供專用空間匯聚相關筆記，積極降低心理負荷
2. **加速 ideation**：在濃縮環境中鼓勵想法快速碰撞與交流
3. **參考工具**：完成一個 MOC 後，專案已完成 80%
4. **導航工具**：連結 vault 其餘部分的可靠入口
5. **提醒工具**：多年後返回，快速記起某主題的完整脈絡
6. **持續演化**：跟隨你的想法成長、成熟、增加複雜性
7. **保持流動**：非破壞性、非限制性，不像資料夾那樣逼迫你用它來存取筆記
8. **無限彈性**：MOC 是**異層級（Heterarchy）**——不同 MOC 可以用不同方式映射相同資訊
9. **空間記憶**：想法作為星座的一部分被記住（古希臘 Method of Loci）

### 7.3 MOC 三階段（Mental Squeeze Point 觸發）

```
Gather (收集) ──→ Collide (碰撞) ──→ Navigate (導航)
  ↓                  ↓                  ↓
 匯聚碎片         強制創意連結        結構化呈現
  焦慮 → 舒緩       探索 → 驚喜        複雜 → 清晰
```

### 7.4 Fluid Frameworks vs Rigid Frameworks

| 維度 | Fluid（MOC/Home/原子筆記） | Rigid（資料夾/目錄） |
| :--- | :--- | :--- |
| 結構 | 非剛性、有機、可按需使用 | 剛性、必須、刪除即消失 |
| 刪除後果 | 筆記依然存在 | 筆記路徑受影響 |
| 強制性 | 非必要支援結構 | 必要支援結構 |
| 組織彈性 | 筆記可同時被多個 MOC 引用 | 筆記只能在一個資料夾中 |

> MOC 是「覆蓋層」——添加相關資訊，但不影響基層筆記。

## 八、防彈筆記法（框架 C / Esor Huang）

### 8.1 核心範式轉移

> **筆記不是倉庫，是行動的副產品。**

| 維度 | 傳統靜態分類 | 防彈筆記（代謝系統） |
| :--- | :--- | :--- |
| 核心目標 | 資訊的分類、保存 | 推動任務進度與產出 |
| 筆記單位 | 來源或時間 | 任務/目標 |
| 演化狀態 | 靜態，寫完後封存 | 動態，隨執行持續修改 |
| 更新頻率 | 極低 | 極高 |
| 失敗模式 | 收藏家謬誤（儲存≠掌握） | 筆記與行動脫節 |

### 8.2 筆記五階段生命週期

```
Inbox → Temp → Project → Resource → Evergreen
  ↓       ↓        ↓         ↓          ↓
 緩衝區  初步轉化   行動指揮    長期參考    智慧結晶
```

| 階段 | 角色 | 演化原則 |
| :--- | :--- | :--- |
| **Inbox** | 接收一切，零門檻 | 僅做初步捕獲 |
| **Temp** | 等待處理的暫時地帶 | 若 48 小時內無行動，評估是否值得保留 |
| **Project** | 單一任務的行動指揮中心 | 任務結束後移出或蒸餾至 Evergreen |
| **Resource** | 長期參考資源庫 | 定期檢視是否需要更新 |
| **Evergreen** | 活得夠久的智慧結晶 | 持續維護，隨個人成長而演化 |

### 8.3 3問3動處理框架

**釐清現況（3問）：**

| 問題 | 意圖 | 回答範例 |
| :--- | :--- | :--- |
| Why/Who | 目的/對象 | 這筆記是為了什麼專案？給誰看？ |
| How | 量化指標 | 如何知道這筆記「完成」了？ |
| What | 阻礙限制 | 什麼阻礙我現在就行動？ |

**決定未來（3動）：**

| 行動 | 做法 |
| :--- | :--- |
| **改寫** | 用自己的語言重新表述（強制的理解檢驗） |
| **連結** | 建立與現有知識網的參照（MOC/相關筆記） |
| **提取** | 轉化為具體待辦事項（下一步行動） |

### 8.4 核心指令

> **更新勝於新增。** 高效防彈系統：每日新增 2-3 則，每日更新 30+ 則舊筆記
> 拒絕收藏家謬誤：儲存資訊不等於掌握資訊。

## 九、Graph View 寫作流程（框架 D / 朱麒）

### 9.1 四種筆記類型

| 類型 | 標籤 | 說明 | 範例 |
| :--- | :--- | :--- | :--- |
| 名詞筆記 | #📖 | 專有名詞解釋 | `#📖 沉沒成本`：已付出且不可回收的成本 |
| 來源筆記 | #📥 | 外界知識（讀書/Podcast/文章）| `#📥《原子習慣》筆記` |
| 關係筆記 | #🔗 | 自己思考後寫下的筆記 | `#🔗 習慣疊加的本質是情境提示` |
| 主題筆記 | #🗺 | Map of Content，能囊括同主題內容 | `#🗺 習慣系統 MOC` |

### 9.2 Graph View 大量寫作五步驟

> **核心論點：寫作卡關是因為參考資料不夠。當我們不必從一張白紙開始時，創作的過程快上許多。**

1. **設定界面**：左側開啟名詞筆記，右側開啟 Graph View（Cmd + 點擊）
2. **Graph View 搜尋關鍵字**：使用 `line:(關鍵字)` 語法，只顯示含有該關鍵字的筆記節點
3. **雙視圖對照**：左側筆記用 Obsidian Query 顯示含關鍵字的段落文字，與右側 Graph View 同步
4. **蒐集碎片**：將有感觸的內容轉貼到「創作筆記」
5. **主動建立連結**：在 Graph View 中發現「有提到但沒有實際連結」的筆記時，主動點擊建立 wikilink

### 9.3 寫作流程

```
碎片化文字 → 整理成完整段落 → 列下文章大綱 → 把段落依序填入大綱
```

```
INPUT → 處理中心(HQ: PKM+PM+Calendar+MOC) → 蒸餾流水線(3問3動/更新>新增) → OUTPUT
```

### 10.2 每週複盤流

```
週日蒸餾任務：
1. 掃描 Inbox → 執行 3問3動（改寫/連結/提取）
2. 更新既有筆記（更新 > 新增）
3. 將 Project 筆記蒸餾至 Evergreen 原子筆記
4. 維護 MOC 連結（評估是否需要 Collide 碰撞）
5. 檢視 Efforts 四種強度，重新分配頻寬
```

### 10.3 工具選擇決策樹

```
需要 All in One？
    ├─ 是 → Notion（覆蓋廣，轉換阻力低）
    └─ 否 → 分工並用
            ├─ 發散處理 → Heptabase 白板 / Obsidian 頁面
            ├─ 收斂整理 → Obsidian/Notion 資料庫
            ├─ 時間管理 → 行事曆工具隔離（不混在 PKM 系統中）
            └─ 連結思考 → Obsidian Graph View
```

## 十二、實踐原則與常見陷阱

### 12.1 進階原則

- **整理發生在執行時**（Organize while doing）
- **拒絕收藏家謬誤**：儲存 ≠ 掌握
- **更新勝於新增**：修改 30+ 筆記 > 新增 3 筆記
- **MOC 是入口**：讓所有資訊最終匯聚至主題地圖
- **發散與收斂平衡**：白板自由探索 ↔ 資料庫結構收斂
- **Efforts 的四種強度**：時常檢視，動態調整頻寬

### 12.2 常見陷阱

| 陷阱 | 對應 |
| :--- | :--- |
| 過早設計完美結構 | 先用預設 KCE，結構需被證明合理性後才自訂 |
| 把收斂工具用於發散階段 | 白板還沒收斂前不要放進資料庫 |
| 只新增不更新 | 每篇新筆記代價 = 更新 10 篇舊筆記 |
| MOC 太早固化 | Gather → Collide → Navigate 順序不能跳過 |
| 笔记者心态（只记录不创作） | 設定輸出目標：用 Graph View 每週寫一篇 |
