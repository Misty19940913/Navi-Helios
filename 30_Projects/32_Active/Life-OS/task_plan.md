## 一、系統定位

### 1.1 Life OS 的核心價值

Life OS 是一套**通用的人生執行嫁接層**，它是你的數位外骨骼。

```
Life OS = KCE 執行框架 + 原子化知識標準 + AI 協作協議

核心目標：
├── 降低認知負荷（KCE 結構）
├── 提升執行彈性（強度管理）
└── 實現知識變現（內容流水線）
```

### 1.2 KCE 核心結構

```
┌─────────────────────────────────────────────────────────────────────────┐
│                   Life OS 三空間執行層 (v4.2)                              │
├───────────────────────┬─────────────────────┬───────────────────────────┤
│  K - Knowledge        │   C - Calendar      │      E - Efforts          │
├───────────────────────┼─────────────────────┼───────────────────────────┤
│ 🧠 知識與理解          │ 🗓️ 時間與日誌        │ 🏋️ 行動與創造             │
│ Library (000-900)     │ Daily Log            │ Vision（橫跨三空間）      │
│ MOCs                  │ Weekly Review        │   ↓                      │
│ Cards                 │ Past Years           │ Goal                     │
│                       │                      │   ↓                      │
│                       │                      │ Effort（四強度）          │
│                       │                      │   ↓                      │
│                       │                      │ Task                     │
└───────────────────────┴─────────────────────┴───────────────────────────┘
```

> **K = Knowledge**（原 Atlas），框架仍稱 ACE。
> **Vision** 橫跨三空間：「我要成為什麼樣的人」滲透在知識/時間/行動中。
> **Effort 四強度**：🔥`On` → ♻️`Ongoing` → 〰️`Simmering` → 💤`Sleeping`。

## 二、系統架構 (v4.2)

### 2.1 四層戰略模型

| 層級 | 頻率 | 元件 | 核心問題 |
| :--- | :--- | :--- | :--- |
| **Vision** | 年度/終身 | 人生願景、領域願景（橫跨三空間） | 我要成為什麼樣的人？ |
| **戰略層** | 年度/季度 | Goal MOC | 我要達成什麼結果？ |
| **戰術層** | 月度/週期 | Effort MOC（On/Ongoing/Simmer/Sleeping） | 我在什麼事上努力？ |
| **執行層** | 每週/每日 | Task, Daily Log, Weekly Review | 我下一步具體行動是什麼？ |

> **Vision 的特殊性**：不隸屬於單一空間。願景陳述出現在 Knowledge（我理解什麼是我想要的）、Calendar（我經歷了什麼影響我的願景）、Efforts（我在創造什麼來實現願景）三個層面。

---

## 三、系統研究順序：Vision → Goal → Effort → Task（日誌殿後）

> **核心原則**：Vision 是源頭（横跨三空間）。Goal 是 Vision → Efforts 的轉化橋樑（戰略層，不隸屬單一空間）。Effort/Task 是 Efforts 空間的執行鏈。Log 是 Calendar 的子元件，依賴 Task 才有鉤子。

```
Vision（願景）
  （不在執行鏈上，但滲透在所有層級）
    ↓
Goal（目標）
    ↓
Effort（努力）  ← 四強度 On/Ongoing/Simmer/Sleeping
    ↓
Task（任務）    ← Log 的鉤子
    ↓
Daily Log（日誌） ← 記錄「實際做了什麼」
```

**為什麼 Log 殿後：**
- Log 沒有 Task 就只是流水帳，覆盤時沒有東西可以回溯
- Task 沒有 Goal 就只是待辦清單，優先級無法判斷
- Goal 沒有 Vision 就只是願望清單，方向無法對齊

## 四、Life OS 新架構設計原則

> **核心立場**：從零開始研究新架構，現有 vault 是「素材參考」不是「遷移目標」。現有資料夾結構（`22_Area`、`30_Projects` 等）是過去的殘留，用完即棄。

### 4.1 設計邏輯：先問「為什麼」，再問「放什麼」

```
第一步：定義系統存在的目的（Why）
    ↓
第二步：三空間各放什麼（What）
    ↓
第三步：每個空間內的子系統如何串聯（How）
    ↓
第四步：元件之間的依賴關係決定研究順序（Which）
```

### 4.2 三空間的定義（KCE）

| 空間 | 核心問題 | 處理什麼 | 子元件 |
|---|---|---|---|
| **Knowledge**（原 Atlas） | 我理解什麼？ | 知識、點子、來源、人物 | Library、MOCs、Cards |
| **Calendar** | 我經歷了什麼？ | 時間日誌、覆盤、回顧 | Daily Log、Weekly Review、Past Years |
| **Efforts** | 我在創造什麼？ | 願景→目標→行動的執行鏈 | Vision（横跨）、Goal（催生）、Effort、Task |

> **Vision 横跨三空間**：願景不屬於單一空間。「我要成為什麼樣的人」是一個滲透在 Knowledge（我理解什麼是值得的）、Calendar（我經歷了什麼塑造了我）、Efforts（我在創造什麼來實現我）中的命題。

### 4.3 執行鏈：Vision → Goal → Effort → Task（Log 是 Calendar 的附屬）

```
┌──────────────────────────────────────────────────────────────────┐
│  Vision（願景）                                                   │
│  「我要成為什麼樣的人」                                             │
│  （横跨 Knowledge / Calendar / Efforts 三空間）                    │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────────┐
│  Goal（目標）     → 由 Vision 催生，橫跨 Area 的戰略規劃          │
│  「我要達成什麼具體結果」                                          │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────────┐
│  Effort（努力）   → 隸屬 Efforts 空間（四強度 On/Ongoing/Simmer）│
│  「我正在什麼事上投入努力」                                         │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────────┐
│  Task（任務）     → 隸屬 Efforts 空間                            │
│  「下一步具體行動是什麼」                                          │
└──────────────────────────────────────────────────────────────────┘
                              ↓ 記錄鉤子
┌──────────────────────────────────────────────────────────────────┐
│  Daily Log（日誌） → 隸屬 Calendar 空間                          │
│  「我實際做了什麼、經歷了什麼」                                     │
└──────────────────────────────────────────────────────────────────┘
```

### 4.4 Area 的新定位：Vision 的副系統

> **澄清**：`22_Area` 原本的定位是「各領域願景」，不是 PARA 的「責任範圍」。

| 舊定位（PARA） | 新定位（Life OS） |
|---|---|
| Area = 無終點的責任範圍 | Area = Vision 的副系統 |
| 需要維護 | 需要具象化（寫成願景陳述） |
| 沒有完成日 | 有領域願景陳述 |

**在 Life OS 新架構中：**
- 每個 `Area` = 一個 **領域 Vision**（如「健康 Vision」、「關係 Vision」、「事業 Vision」）
- `Area Vision` 匯聚成一個「人生整體 Vision」
- Area Vision 不是「責任」而是「方向」

### 4.5 為什麼廢除現有資料夾遷移

```
現有 vault 的價值：
    是「素材」，不是「系統」
    是「過去經驗的沉澱」，不是「未來架構的藍圖」

正確使用方式：
    從 vault 提取有用的概念（如 ACE 的 Effort 四強度）
    ↓
    研究新架構
    ↓
    新架構有完整邏輯後，才決定哪些舊資料需要蒸餾進來
    ↓
    舊資料夾結構（22_Area、30_Projects）是「燃燒的燃料」，
    不是「要保留的建築」
```

## 八、系統架構與研究邏輯

> 先有架構，才有順序。本文件是「如何研究系統」，不是「如何建構系統」。研究完成後的實作路徑，視研究產出再另文定義。

### 架構邏輯

```
系統定位（Why）
 └─ KCE 核心結構（What — 三空間各放什麼）
     └─ 四層戰略模型（How — Vision/Goal/Effort/Task/Log 怎麼串聯）
         └─ 研究順序（Which — 先研究哪個元件）
```

### 為什麼倒數第二研究 Library（T7）

```
Library 是 Knowledge 的導航入口
 Knowledge 是 PKM 處理完後的「知識沉澱區」

 未研究清楚就建目錄 → 目錄是空的
 先研究後再建目錄 → 目錄是自然生長的
```

### 為什麼最後研究 AI 協作（T10）

```
AI 協作是加速器，不是核心元件
 系統結構還沒研究清楚就讓 AI 操作 → 加速破壞
 系統結構研究清楚後再讓 AI 加入 → 加速建設
```
