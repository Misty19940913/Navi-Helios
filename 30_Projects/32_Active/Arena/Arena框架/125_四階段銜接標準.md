# 125_四階段銜接標準

> Arena 四個 Stage 之間的銜接邏輯與驗證標準
> 產出時間：2026-04-01

---

## 四階段銜接總覽

```
Stage 1: 認識自己
    ↓ 銜接標準 1
Stage 2: 規劃目標
    ↓ 銜接標準 2
Stage 3: 策略規劃
    ↓ 銜接標準 3
Stage 4: 迭代執行
    ↓ 銜接標準 4
Stage 1（新循環）
```

---

## 銜接標準 1: Stage 1 → Stage 2

### 顯著變化定義

什麼算「Stage 1 完成」？

```
門檻 A（最低標準）：必須全部滿足
□ 六商（FQ/HQ/EQ/PQ/AQ/XQ）全部有等級 + 階段標註
□ 生活方式原型識別完成
□ 至少 1 個 x.1 失調信號被識別

門檻 B（顯著變化）：至少滿足一項
□ 至少 1 個商數從 x.1 進展到 x.2 或 x.3（失調緩解）
□ 生活方式原型从失衡原型變為平衡原型
□ 跨商傳導路徑从 2 條以下增加到 3 條以上
□ 用戶主觀描述與客觀評估吻合度 > 70%（自述 vs AI 評估）

門檻 C（失敗/退行信號）：滿足任一項應暫停 Stage 2
□ 2+ 個商數處於 x.1 且用戶否認失調存在
□ 用戶拒絕識別任何錯誤轉變信號
□ 三週內未完成每日覺察記錄（好時光/壞時光）
```

### Stage 1 產出要求

完成 Stage 1 後，必須產出：

| 產出 | 格式 | 驗證問題 |
|------|------|----------|
| 六商現況評估 | 等級 + 階段 | 「每個商數的現況是否清晰？」 |
| 生活方式原型 | 原型名稱 | 「原型的失衡點是否識別？」 |
| 錯誤轉變訊號 | 有/無 + 描述 | 「是否有需要注意的假成長？」 |
| 跨商傳導 | 影響關係圖 | 「商數間的影響是否清楚？」 |
| 優先發展商數 | 商數 + 理由 | 「優先順序是否有邏輯？」 |

### Stage 2 輸入要求

進入 Stage 2 前必須確認：

| 輸入 | 來源 | 確認問題 |
|------|------|----------|
| 現況商數等級 | Stage 1 | 「商數等級是否準確？」 |
| 落差分析 | Stage 1 | 「現況與理想的差距是否清楚？」 |
| 發展優先順序 | Stage 1 | 「為什麼這個順序？」 |
| 主要阻礙 | Stage 1 | 「Stage 1 有沒有發現關鍵阻礙？」 |

### 銜接驗證問題

| 驗證問題 | 回答 | 結果 |
|----------|------|------|
| Stage 1 的自我認知是否完整？ | 是/否 | |
| Stage 1 有沒有發現關鍵失衡？ | 有/無 | |
| 發展優先順序是否有邏輯？ | 是/否 | |
| 是否有足夠資訊進入目標設定？ | 是/否 | |

### 銜接失敗的處理

| 失敗類型 | 處理方式 |
|----------|----------|
| Stage 1 產出不完整 | 返回補充 |
| 自我認知有盲點 | 增加訪談或測試 |
| 優先順序不明 | 重新分析 |

## 銜接標準 2: Stage 2 → Stage 3

### 顯著變化定義

什麼算「Stage 2 完成」？

```
門檻 A（最低標準）：必須全部滿足
□ 大目標設定完成（含等級需求對照）
□ 六商差距分析表完成
□ 至少 1 個進化台階定義清晰

門檻 B（顯著變化）：至少滿足一項
□ 至少 1 個商數的差距從「不可達」變為「可達」（差距縮小 0.3+）
□ 用戶主觀意願與客觀條件匹配度提升
□ 出現新的路徑替代方案（本來只有 1 條，現在有 2+ 條）
□ 資源清單新增關鍵資源（本來缺的資源現在有了）

門檻 C（失敗/退行信號）：滿足任一項應重新審視 Stage 2
□ 目標設定後連續兩週沒有任何進展
□ 用戶對目標的熱情明顯下降（自述「沒感覺了」）
□ 識別出原來的前置條件其實不滿足
```

### Stage 2 產出要求

完成 Stage 2 後，必須產出：

| 產出 | 格式 | 驗證問題 |
|------|------|----------|
| 大目標設定 | 目標描述 + 等級需求 | 「目標是否具體可衡量？」 |
| 前置條件審核 | 商數等級需求對照 | 「具備起步條件嗎？」 |
| 差距分析 | 六商差距分析表 | 「每個商數的差距是否清楚？」 |
| 進化台階 | 台階 1/2/3... | 「台階是否可達成？」 |
| 路徑圖 | 時間軸 + 里程碑 | 「路徑是否清晰？」 |
| 原型驗證設計 | 實驗設計 | 「驗證方式是否可行？」 |
| 資源盤點 | 四類資源清單 | 「關鍵資源是否充足？」 |
| 支持系統 | 支持者清單 | 「支持系統是否完整？」 |

### Stage 3 輸入要求

進入 Stage 3 前必須確認：

| 輸入 | 來源 | 確認問題 |
|------|------|----------|
| 目標維度與等級 | Stage 2 | 「目標是否合理？」 |
| 進化台階 | Stage 2 | 「台階是否可行？」 |
| 資源清單 | Stage 2 | 「資源是否足夠？」 |
| 支持系統 | Stage 2 | 「支持是否到位？」 |

### 銜接驗證問題

| 驗證問題 | 回答 | 結果 |
|----------|------|------|
| 目標是否具體可衡量？ | 是/否 | |
| 前置條件是否滿足？ | 是/否 | |
| 差距是否在可接受範圍？ | 是/否 | |
| 路徑是否有替代方案？ | 有/無 | |
| 資源是否基本充足？ | 是/否 | |

### 銜接失敗的處理

| 失敗類型 | 處理方式 |
|----------|----------|
| 目標太模糊 | 返回重新設定 |
| 前置條件不足 | 先發展落後維度 |
| 差距過大 | 調整目標或增加資源 |
| 路徑不可行 | 重新設計路徑 |

## 銜接標準 3: Stage 3 → Stage 4

### 顯著變化定義

什麼算「Stage 3 完成」？

```
門檻 A（最低標準）：必須全部滿足
□ 30天行動計劃覆蓋所有六商
□ PDCA 循環模板建立完成
□ Checkpoint（週/月）設定明確

門檻 B（顯著變化）：至少滿足一項
□ 第一週行動完成率 > 80%
□ 至少 1 個商數出現可測量的初期變化
□ 失敗分類系統建立，用戶能區分「努力失敗」vs「策略失敗」

門檻 C（失敗/退行信號）：滿足任一項應暫停 Stage 4
□ 第一週行動完成率 < 30%（執行意願存疑）
□ 用戶完全無法區分「累」和「停滯」的身體信號
□ 沒有任何支持系統（行動計劃缺乏外部支持）
```

### Stage 3 產出要求

完成 Stage 3 後，必須產出：

| 產出 | 格式 | 驗證問題 |
|------|------|----------|
| 行動類型定義 | 習慣/任務/原型/里程碑 | 「行動類型是否清楚？」 |
| PDCA 循環 | 每週模板 | 「循環機制是否完整？」 |
| Checkpoint 設定 | 週/月/季/年 | 「評估點是否合理？」 |
| 失敗分類系統 | 四類失敗 + 記錄 | 「失敗處理機制是否清楚？」 |
| 30 天行動計劃 | 週計劃 + 評估 | 「第一個月是否可執行？」 |
| 商數追蹤表 | 六商追蹤 | 「追蹤指標是否合理？」 |
| 調整觸發條件 | 四種觸發 | 「觸發條件是否清楚？」 |

### Stage 4 輸入要求

進入 Stage 4 前必須確認：

| 輸入 | 來源 | 確認問題 |
|------|------|----------|
| 行動計劃 | Stage 3 | 「第一期行動是否明確？」 |
| PDCA 模板 | Stage 3 | 「循環機制是否理解？」 |
| Checkpoint 設定 | Stage 3 | 「評估點是否合理？」 |
| 觸發條件 | Stage 3 | 「什麼情況要調整？」 |

### 銜接驗證問題

| 驗證問題 | 回答 | 結果 |
|----------|------|------|
| 第一期行動是否明確？ | 是/否 | |
| PDCA 機制是否理解？ | 是/否 | |
| Checkpoint 是否有時間？ | 是/否 | |
| 調整觸發是否清楚？ | 是/否 | |

### 銜接失敗的處理

| 失敗類型 | 處理方式 |
|----------|----------|
| 行動不明確 | 回到 Stage 3 補充 |
| PDCA 機制不理解 | 重新說明或簡化 |
| Checkpoint 太少 | 增加評估點 |
| 觸發條件模糊 | 重新定義觸發標準 |

## 銜接標準 4: Stage 4 → Stage 1（新循環）

### 顯著變化定義

什麼算「Stage 4 完成可以進入新循環」？

```
門檻 A（最低標準）：必須全部滿足
□ 完成至少 1 個完整 PDCA 循環（4 週以上執行記錄）
□ Checkpoint 評估完成（商數進化評估）
□ 循環總結寫出（目標達成情况 + 關鍵學習）

門檻 B（顯著變化）：至少滿足一項
□ 至少 1 個商數提升 0.3+ 等級
□ 至少 1 個 x.1 商數轉變為 x.2 或 x.3
□ 生活方式原型發生可觀察的正面轉變
□ 新的循環方向明確，且有別於上一輪

門檻 C（失敗/退行信號）：滿足任一項應先處理再開始新循環
□ 所有商數均無進步且無退化（等於停滯）
□ 目標完全沒有推進（執行率 = 0%）
□ 用戶表達創傷或burnout需要先休息
□ 支持系統完全崩塌（沒有任何外部支持）
```

### Stage 4 產出要求

完成 Stage 4 後，必須產出：

| 產出 | 格式 | 驗證問題 |
|------|------|----------|
| PDCA 執行記錄 | 完整循環記錄 | 「執行過程是否完整記錄？」 |
| Checkpoint 評估 | 商數進化評估 | 「商數是否有進步？」 |
| 失敗與教訓 | 失敗分類 + 學習 | 「從失敗中學到了什麼？」 |
| 調整記錄 | 調整事件清單 | 「做了哪些調整？」 |
| 循環總結 | 新循環準備 | 「是否準備好開始新循環？」 |

### 新循環 Stage 1 輸入要求

進入新循環 Stage 1 前必須確認：

| 輸入 | 來源 | 確認問題 |
|------|------|----------|
| 更新後的商數等級 | Stage 4 | 「經過這輪，商數有什麼變化？」 |
| 目標達成情况 | Stage 4 | 「設定的目標完成了多少？」 |
| 關鍵學習 | Stage 4 | 「這輪學到了什麼重要的？」 |
| 調整觸發原因 | Stage 4 | 「為什麼要開始新循環？」 |

### 觸發 Stage 1（新循環）的驗證問題

| 驗證問題 | 回答 | 結果 |
|----------|------|------|
| 是否符合觸發條件？ | 是/否 | |
| 上一輪學習是否整理？ | 是/否 | |
| 是否自願開始新循環？ | 是/否 | |
| 是否有明確的新重點？ | 是/否 | |

### 銜接失敗的處理

| 失敗類型 | 處理方式 |
|----------|----------|
| 執行記錄不完整 | 補充記錄 |
| 商數無進步 | 分析原因，可能需要大調整 |
| 觸發條件模糊 | 重新確認觸發原因 |
| 不自願開始新循環 | 尊重意願，延後開始 |

