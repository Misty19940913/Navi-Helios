# Stage 3：策略規劃

> 這是 Arena 課程的第三個階段。你的任務是：基於 Stage 2 的路徑規劃，產出具體的可執行行動方案，並建立持續迭代的執行系統。
>
> **核心問題：我如何到達？**

---

## Stage 3 總覽

### 目標

將 Stage 2 的願景方向和路徑規劃，轉化為每週、每天可執行的行動，並建立 PDCA 循環系統，讓你在執行中學習、在學習中調整。

### 與 Stage 2 的銜接

Stage 2 回答了「我要去哪裡」的問題，並給了你一份路徑規劃圖。Stage 3 的任務，是把這份規劃圖翻譯成：

- **誰**要在 **什麼時間** 做 **什麼行動**
- **如何追蹤**執行的效果
- **如何判斷**需要調整或重新開始

### 流程

```
Step 1 → 資源盤點（我有什麼可以用的？）
Step 2 → 行動分類與 PDCA 設計
Step 3 → 失敗分類系統建立
Step 4 → 產出 30 天行動計劃
Step 5 → 建立每日/每週 PDCA 操作系統
Step 6 → 建立商數追蹤與等級進化評估
Step 7 → 建立階段調整機制
Step 8 → AI 訪談對齊（取得客觀視角）
```

### 產出

一份完整的「30 天行動計劃」+ 一套可持續運作的「PDCA 執行系統」，作為 Stage 4（迭代執行）的基礎。

---

## 3.0 前置說明

### 為什麼要盤點資源？

多數人在規劃行動時，常見的錯誤是：只規劃「要做的事」，忽略了「要用什麼資源做」。

如果你的行動計劃需要的資源你根本沒有，計劃就會變成空轉。資源盤點的目的，是讓你在規劃行動之前，先誠實地面對自己擁有什麼、缺少什麼。

### 為什麼要 PDCA？

傳統的目標執行邏輯是：設定目標 → 執行到底。但現實中，環境會變化、假設會失準、自己的能力也會成長或停滯。

PDCA 的精神是：**目標是起點，不是終點。** 每一次執行，都是一次學習的機會；每一次檢查，都是下一次計劃的基礎。

## 3.3 合作夥伴網絡

### 合作夥伴類型

| 類型 | 說明 | 對 Arena 的價值 |
|------|------|----------------|
| **策略合作** | 共同開發市場或產品 | 擴大影響力 |
| **互補型合作** | 填補你的能力缺口 | 完整服務交付 |
| **平台合作** | 分銷或曝光管道 | 獲取流量 |
| **聯盟合作** | 互相推廣、資源置換 | 資源共享 |

### 合作評估問題

在考慮任何合作之前，先問這四個問題：

| 問題 | 為什麼重要 |
|------|-----------|
| **互補性**：對方的優勢能否補足我的劣勢？ | 否則合作沒有實際價值 |
| **價值觀一致性**：我們的核心價值觀是否吻合？ | 否則遲早會衝突 |
| **利益分配**：合作的利益如何分配？ | 否則會埋下分裂的種子 |
| **退出機制**：如果合作不順利，如何優雅退出？ | 否則會被困住 |

## 3.5 行動分類與 PDCA 設計

### 四種行動類型

不是所有行動都一樣重要。Stage 3 的行動可以分為四種類型：

| 行動類型 | 說明 | 範例 | Arena 對應 |
|----------|------|------|-----------|
| **日常習慣** | 每日/每週例行執行 | 每日運動、晨間反思 | 商數維持 |
| **階段任務** | 達成台階目標的關鍵行動 | 開發課程、建立客戶名單 | 進化台階 |
| **原型測試** | 驗證假設的小實驗 | 測試定價、發布最小可行產品 | 驗證性行動 |
| **里程碑事件** | 重大轉折點 | 發布會、發表報告 | 目標達成 |

### PDCA 迴圈原理

PDCA 是一個持續循環的系統：

```
Plan（計劃）
  ↓  我這週要做什麼？為什麼？
Do（執行）
  ↓  實際執行，並記錄發生了什麼
Check（檢查）
  ↓  和計劃比，我達成了什麼？沒達成什麼？
Act（調整）
  ↓  下週要做什麼不一樣？
```

**關鍵原則：每一次 Check，都是下一次 Plan 的基礎。**

### Checkpoint 評估點

Checkpoint 是讓你停下來、重新評估方向的節點。

| Checkpoint | 頻率 | 評估商數 | 決策 |
|------------|------|----------|------|
| **週檢查** | 每週 | 當週進度 | 調整下週行動 |
| **月檢查** | 每月 | 月度目標 | 調整月度計劃 |
| **季評估** | 每季 | 階段目標 | 調整進化台階 |
| **年回顧** | 每年 | 年度願景 | 重新規劃 |

### Checkpoint 評估問題

**週檢查：**

| 追蹤商數 | 問題 |
|------|------|
| 進度 | 「這週完成了什麼？與目標相比如何？」 |
| 能量 | 「這週的能量狀態如何？什麼帶來能量？什麼消耗能量？」 |
| 學習 | 「這週學到了什麼？」 |
| 調整 | 「下週有什麼需要調整？」 |

**月檢查：**

| 商數 | 問題 |
|------|------|
| HQ/XQ/EQ/PQ/AQ/FQ進步 | 「這個月每個商數有進步嗎？」 |
| 台階進度 | 「離下一個台階還有多遠？」 |
| 假設驗證 | 「原型測試的結果如何？」 |
| 計劃調整 | 「月度計劃需要調整嗎？」 |

## 3.6 失敗分類系統

### 為什麼要建立失敗分類？

多數人對失敗的理解是：「失敗了 = 我不行」。這是一種「固定型思維」。

Arena 的失敗分類系統，是要建立一種新的理解框架：**失敗是數據，不是身份**。每一次失敗，都是系統反饋的機會。

### 四種失敗類型

| 失敗類型 | 說明 | 處理方式 |
|----------|------|---------|
| **簡單錯誤** | 可立即修正的疏忽 | 立即修正，繼續前進 |
| **弱點觸發** | 你的個人限制被引發 | 覺察它，並建立應對機制 |
| **成長機會** | 有明確解決方案的挑戰 | 學習並克服 |
| **系統失敗** | 外部因素導致的失敗 | 調整你的系統 |

### 失敗記錄模板

```
【失敗事件】
日期：
事件描述：

【失敗分類】
類型：_______________

【根本原因分析】
-

【學到的教訓】
-

【應用於未來】
下次遇到類似情況，我會：
```

### 失敗免疫力建立步驟

| 步驟 | 行動 |
|------|------|
| 1. 記錄失敗 | 每次失敗後立即記錄 |
| 2. 重構框架 | 將失敗視為學習數據，而不是身份的定義 |
| 3. 分享受控 | 與信任的人分享失敗經驗 |
| 4. 建立支持 | 知道有人在背後支持你面對失敗 |

## 3.11 階段調整機制

### 什麼時候需要調整？

執行過程中，難免會遇到需要調整的情況。不是所有的調整都一樣——調整分為三個層級：

| 調整類型 | 觸發條件 | 調整幅度 | 回到哪裡 |
|----------|----------|---------|---------|
| **例行調整** | 每週 Checkpoint 發現未達標 | 微調 | 繼續 Stage 3 |
| **方向調整** | 連續 2-3 月未達成目標 | 中幅調整 | 回到 Stage 2 重新設計路徑 |
| **框架調整** | 價值觀或目標根本性轉變 | 大幅調整 | 回到 Stage 1 重新開始 |

### 觸發回 Stage 1 的判斷矩陣

以下任一條件觸發時，考慮回到 Stage 1 重新開始整個循環：

| 條件類型 | 具體觸發條件 | 優先級 |
|----------|-------------|--------|
| **商數崩潰** | 任何商數持續 2 週以上呈現停滯或退化狀態 | 高 |
| **等級退化** | 任何商數等級下降超過 0.3 | 高 |
| **目標失效** | 原有目標已經完全失去意義 | 高 |
| **價值觀轉變** | 對工作/生活/關係的核心價值觀發生重大轉變 | 中 |
| **外部巨變** | 事業/家庭/健康發生根本性改變 | 高 |
| **成長停滯** | 連續 3 個月無任何商數進步 | 中 |
| **長期倦怠** | 持續 1 個月以上的 burnout 狀態 | 高 |

### 觸發判斷問題清單

每月底，回答這個清單：

| 問題 | 是 | 否 |
|------|----|----|
| 任何商數連續 2 週呈現停滯或退化狀態？ | → 警訊 | → 正常 |
| 任何商數等級下降超過 0.3？ | → 警訊 | → 正常 |
| 原有目標已經完全失去意義？ | → 警訊 | → 正常 |
| 價值觀發生重大轉變？ | → 關注 | → 正常 |
| 外部環境發生根本性改變？ | → 警訊 | → 正常 |
| 連續 3 個月無進步？ | → 警訊 | → 正常 |
| 持續 1 個月以上倦怠？ | → 警訊 | → 正常 |

### PDCA 執行常見陷阱

| 陷阱 | 說明 | 避免方式 |
|------|------|---------|
| **只做不檢查** | 只行動不回顧 | 強制每日/每週 Check |
| **過度計劃** | 計劃太多執行太少 | 設定每日行動上限（最多 3 件）|
| **忽略失敗** | 不記錄不分析失敗 | 建立失敗記錄系統 |
| **單一商數** | 只關注 FQ 忽略其他 | 每週檢查所有六個商數 |
| **目標過大** | 目標超出能力範圍 | 分解為每週可執行的小步驟 |

## 3.12 AI 訪談對齊

### 為什麼需要 AI 訪談？

行動計劃出來之後，你可能已經有了一些盲點：
- 計劃太過樂觀，低估了執行的難度
- 資源分配不夠現實
- 失敗時不知道問題出在哪裡

AI 訪談提供一個外部視角，檢驗你的行動計劃是否真的可行。

### AI 訪談觸發條件（教練/Agent 專用）

**① 當學員的行動計劃超過其可用資源時：**
> 「你的行動計劃需要每週投入 X 小時，但你之前說你每週只有 Y 小時。這個缺口怎麼解決？」

**② 當學員的 Checkpoint 顯示連續未達標時：**
> 「你已經連續 X 週沒有達成週目標了。你覺得根本原因是什麼？」

**③ 當學員在失敗記錄中重複出現同類失敗時：**
> 「你在失敗記錄裡已經第三次出現同一個問題了。這個模式告訴你什麼？」

**④ 當學員的調整記錄顯示調整頻率過高時：**
> 「你這個月已經做了 X 次重大調整。這個頻率是否說明你的規劃本身有問題？」

**⑤ 當學員的商數追蹤顯示單一商數持續落後時：**
> 「你的 HQ 商數已經連續 X 週沒有改善了。這是你需要優先處理的信號嗎？」

### 如何使用這個區塊

完成 30 天行動計劃之後，帶著這份計劃進入 AI 訪談。

**訪談完成後，將 AI 的反饋與你的計劃並列對比：**

```
項目          | 我的計劃            | AI 反饋              | 需要調整的地方
--------------|--------------------|----------------------|------------------
資源分配      |                    |                      |
行動優先順序  |                    |                      |
失敗應對策略  |                    |                      |
商數平衡      |                    |                      |
調整頻率      |                    |                      |
```

Stage 3「策略規劃」的核心任務已結束。

## 附錄一：PDCA 執行原則

| 原則 | 說明 |
|------|------|
| **每日優先** | 每天結束前完成當日的 Check |
