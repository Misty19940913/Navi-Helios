# Navi Helios 12週主權系統 3.0 完整計畫書

## 一、總體架構與編號邏輯

本系統採用「權力下行」邏輯，數字越小權限越高。

| 編號 | 層級名稱 | 職能定位 | 核心問題 |
|------|---------|---------|---------|
| 100 | 夢想與願景 | 北極星/最高主權 | 我想成為什麼樣的人？ |
| 010 | 目標控制層 | 戰略中樞/12週循環 | 這12週結束時，什麼成果能證明我沒浪費時間？ |
| 090 | KPI 儀表板 | 數據審判/成果追蹤 | 我產出了什麼可證明的現金、影響力或能力？ |
| 001 | 執行層 | 唯一戰場/集權執行 | 今天的 3 個「不可失敗 (MITs)」完成了嗎？ |
| 200 | 專案領域 | 執行骨幹/里程碑 | 為了達成 KPI，目前的專案進度到哪？ |
| 300 | 知識資產 | 彈藥庫/專案燃料 | 這些資訊是否讓我的行動變快？ |
| 400 | 創作區 | 擴音器/影響力變現 | 誰看見了我？誰因此信任我？ |
| 000 | 捕獲區 | 過濾器/7天審判 | 這是黃金還是垃圾？ |
| 500-700 | 支援與存檔 | 後勤部/公墓 | 如何降低摩擦？哪些東西該退出舞台？ |

## 二、各層級細節說明

### 1. 層級一：100 夢想（最高主權）

- **功能定位**：長期願景與價值觀。為 010 提供「為什麼要做」的權力來源。
- **運作規則**：
  - **110 願景規劃**：定義 3-5 年的長期目標。
  - **轉化機制**：每個 12 週開始時，必須對齊願景，不符合願景的目標不予通過。
  - **核心機制**：每年更新一次，每季回顧一次。

### 2. 層級二：010 目標控制層（12週戰略）

- **功能定位**：將願景轉化為「可被裁決」的 12 週目標。
- **核心機制**：
  - **Week 0（規劃週）**：設定 3 個核心目標（權重分配如 50%/30%/20%）。
  - **210 每週計分卡**：本層級最重要的文件。將 12 週目標拆解為本週的 WAP（Weekly Action Plan）。
- **運作規則**：目標一旦設定，12 週內不可隨意增加新目標。

### 3. 層級三：090 KPI 儀表板（成果數據）

- **功能定位**：用數據打臉或證實你的努力。
- **KPI 三大維度**：
  - **現金流 (Cashflow)**：實際到帳金額。
  - **影響力 (Influence)**：新增訂閱、追蹤、邀約數。
  - **能力建設 (Capability)**：完成的主題研究、習得的技能。
- **運作規則**：每週日晚上強制更新。

### 4. 層級四：001 執行層（任務全集權）

- **功能定位**：系統唯一的行動入口。所有任務（無論來自哪個專案）統一在此。
- **關鍵機制**：
  - **MITs 鐵律**：每日嚴格限制 3 個最重要任務。
  - **解釋義務 (Accountability Obligation)**：若任務過期或 MIT 未達成，必須在 `過期任務追蹤.md` 勾選原因：
    - 目標錯誤
    - 拆解不足
    - 外部干擾
    - 能力不足
    - 不允許沈默延期。
- **運作規則**：所有任務必須有 `YYYY-MM-DD` 到期日。

### 5. 層級五：300 知識（執行導向燃料）

- **功能定位**：知識不是收藏，是為了讓 200/400 區運作得更快。
- **核心機制**：
  - **310 專案知識**：與 200 區專案一對一綁定。
  - **30天代謝規則**：每 12 週覆盤時，檢查知識是否曾連結至 200（任務）或 400（創作）。
  - **處置**：未被呼叫的知識直接進入 700 存檔，保持 300 區的「高純度」。

### 6. 層級六：400 創作區（變現漏斗）

- **功能定位**：內容產出、影響力放大。
- **閉環機制**：發布後的內容需回流至 300 區提取「金句」或「方法論」，完成創作到知識的反哺。

## 三、層級間的權力與資訊流動

### 權力流 (Power Flow)

```
100 願景 → 010 目標 → 200 專案 → 001 任務
```

> **註**：如果 001 的任務不能向上追溯到 100，這件任務就不具備正當性。

### 支援流 (Support Flow)

```
000 捕獲 → 300 知識 → 400 創作 → 090 KPI
```

### 退出流 (Exit Flow)

```
010 復盤 → 800 存檔
```

## 四、命名規則與版本控制

### 命名格式

- **任務/日誌**：`YYYY-MM-DD-描述`（例如：`2026-01-07-產品企劃`）
- **專案/知識**：`編號-主題`（例如：`200-個人品牌專案`）

### 版本控制

- **原則**：不改檔名
- **使用 Properties (YAML) 管理**：

```yaml
version: 1.2
status: 🔴待處理/🟢執行中/✅完成
deadline: 2026-01-15
kpi_link: [[090-本期儀表]]
```

## 五、時間循環運作邏輯

### 1. 每日循環（Tactical）

- **AM 規劃**：在 `001/今日作戰.md` 設定 3 個 MITs（從 `210 計分卡` 提取）。
- **PM 清算**：填寫覆盤，若 MIT 失敗，執行「解釋義務」。

### 2. 每週循環（Weekly Cadence）

- **週日晚上**：
  - 更新 `090 KPI 儀表板` 數據。
  - 統計 `210 每週計分卡` 執行率（目標 $\ge 85\%$）。
  - 根據 `300 專案` 的里程碑，規劃下週任務至 `210`。

### 3. 12週循環（Strategic）

- **規劃期 (Week 0)**：設定新 12 週目標，重整 010。
- **衝刺期 (Week 11-12)**：放棄次要任務，全力衝刺 090 中的 KPI。
- **復盤期 (Week 13)**：執行「系統大掃除」，清空 000，封存 300/400 區過時筆記。

## 七、關鍵機制的深度評估

### 解釋義務的殘酷性

這是本系統的核心。它阻止了你「假裝在忙」。當你必須寫下「因為我能力不足」或「這個目標根本是錯的」時，你會被迫調整戰略。

### 知識的專案綁定

徹底解決「收藏家」心態。每一份進入 300 的知識，都必須在 200 專案中找到它的「消費者」。

### 12週的緊迫感

一年有 4 次「年底」的感覺，這會讓你的產出頻率提高 300%。
